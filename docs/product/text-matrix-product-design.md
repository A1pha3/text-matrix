## 1. 产品概述

### 1.1 产品名称

Text Matrix

### 1.2 一句话定位

Text Matrix 是一个面向中文开发者的 AI Agent、AI 开发工作流与开源工具知识库。

### 1.3 产品本质

Text Matrix 不是单纯的博客，也不是只追热点的 AI 资讯站。它的产品目标，是把高时效内容、深度技术内容、学习路径和工具判断组织成一个可检索、可复用、可持续更新的中文内容系统。

从产品形态上看，它更接近以下三者的结合：

- 技术媒体：持续覆盖 AI Agent、Coding Agent、开源工具与行业变化
- 知识库：通过专题页、分类页、搜索与内链把内容串成体系
- 内容产品：通过首页、专题页、文章页和转化入口形成稳定用户路径

### 1.4 当前阶段判断

当前站点已经完成从“内容仓库”到“初步产品化内容站”的第一阶段转换，主要特征包括：

- 首页已经不再是纯时间流，而是带有定位、分流、专题入口和 CTA 的入口页
- 已有 3 个核心专题页：
  - `/ai-agent/`
  - `/coding-agent/`
  - `/open-source-ai-tools/`
- 文章页已经补齐继续阅读与基础转化区块
- 搜索页、分类页、标签页和栏目页已具备探索型入口
- 发布流程已经加入未来时间校验与关键页面校验

从更完整的产品视角看，当前仍处于 `内容系统化早中期`。下一阶段的重点不是继续增加入口数量，而是提高核心入口的解释力、分流能力和可衡量性。

## 2. 产品目标

### 2.1 总目标

把 Text Matrix 从一个高质量技术内容站，推进为围绕 AI Agent 与开发工作流的中文知识入口。

### 2.2 当前阶段北极星

当前阶段最重要的产品目标是：

```text
让新用户能快速理解站点定位，并进入一个可持续阅读的主题路径。
```

这一目标优先于短期订阅、商业合作和泛流量增长。站点应先证明自己能组织内容、解释主题、引导阅读，再逐步强化更深的转化。

### 2.3 分阶段目标

#### 近期目标

- 强化首页与专题页的导航能力
- 让新用户更快理解站点定位与阅读入口
- 提高从首页进入专题页、从专题页进入文章页的效率
- 降低快讯内容对主品牌心智的稀释
- 建立最小可用的产品指标口径

#### 中期目标

- 建立专题页驱动的内容结构，而不是单篇文章驱动
- 形成可复用的“内容发现 -> 系统阅读 -> 轻转化”路径
- 提高站内深度阅读率与多页访问率
- 建立可复用的专题扩张机制

#### 长期目标

- 把站点沉淀成中文 AI Agent / Coding Agent / 工具工作流领域的稳定知识入口
- 在内容资产之外，形成方法论资产和工作流资产
- 为订阅、合作、咨询、产品化服务留出明确接口

## 3. 用户与场景

### 3.1 当前优先用户

当前阶段优先服务 A 类和 C 类用户。B 类用户具有长期价值，但不应在首页首屏和专题结构中抢占第一优先级。

| 用户类型 | 当前优先级 | 典型入口 | 核心需求 | 首个动作 | 转化目标 |
| --- | --- | --- | --- | --- | --- |
| A 类：开发者 | P0 | 首页、搜索、技术文章 | 判断工具价值，建立工作流认知 | 进入专题页或搜索具体工具 | 深度阅读、多页访问 |
| C 类：AI 工具重度使用者 | P0 | 搜索、外部文章、专题页 | 获取对比、边界和真实项目经验 | 阅读高价值文章并回流专题 | 继续阅读、收藏、联系 |
| B 类：独立开发者 / 技术创作者 | P1 | 首页、About、专题页 | 建立产出系统和长期方法论 | 阅读方法论内容 | 联系、合作、订阅 |
| 次级用户 | P2 | 搜索、快讯、社交分享 | 快速了解新概念或新工具 | 阅读单篇内容 | 回访或搜索 |

### 3.2 核心用户画像

#### A 类用户：开发者

特征：

- 对 AI 工具和开发效率提升有较强兴趣
- 愿意读中长篇技术内容
- 更关注工具边界、工作流和工程实践

主要需求：

- 快速判断某个工具是否值得用
- 理解 Agent、MCP、Skills、Browser automation 等概念之间的关系
- 找到更稳定、更长期的开发工作流

#### B 类用户：独立开发者 / 技术创作者

特征：

- 更关注工具选择、工作流搭建、效率提升和长期积累
- 除技术深度外，也会关心商业化、内容生产与方法沉淀

主要需求：

- 选择合适的 AI 工具栈
- 理解如何把 AI 工具接进自己的产出系统
- 获得专题化、方法论化内容，而不是零散教程

#### C 类用户：AI 工具重度使用者

特征：

- 已经在使用 Claude Code、Codex、Cline、Roo Code 或类似产品
- 对 AI Agent 和 Coding Agent 的实践细节更敏感

主要需求：

- 更深的对比、选型与边界判断
- 更贴近真实项目的工作流经验
- 更高质量的参考文章和工具索引

### 3.3 核心使用场景

| 场景 | 用户问题 | 应提供的入口 | 成功信号 |
| --- | --- | --- | --- |
| 第一次进入站点 | 这是一个什么站？我该从哪开始？ | 首页首屏定位、专题入口、搜索入口 | 用户在 5 到 15 秒内进入专题或搜索 |
| 围绕主题系统学习 | 我想理解 AI Agent / Coding Agent / 开源 AI 工具 | 专题页、精选文章、阅读路径 | 用户从专题页进入至少 1 篇文章 |
| 读完文章后继续探索 | 还有没有相关内容？这篇属于哪条线？ | 文章页继续阅读、专题回流、相关文章 | 用户从文章页进入专题或另一篇文章 |
| 明确问题或工具查找 | 某个工具、概念、项目在哪里？ | 搜索、标签、分类、专题页索引 | 用户使用搜索或标签进入内容 |
| 建立长期关注关系 | 这个站是谁做的？后续怎么跟进？ | About、Contact、后续订阅入口 | 用户访问 About / Contact 或订阅入口 |

## 4. 产品定位与价值主张

### 4.1 不做什么

为了让产品方向更清晰，Text Matrix 不以以下形态为主要目标：

- 不做泛 AI 新闻站
- 不做纯个人日记式博客
- 不做只追热点、不沉淀知识结构的内容流
- 不做只依赖单篇爆文的流量站
- 不在内容结构未成熟前强行推进重商业化转化

### 4.2 核心价值主张

Text Matrix 提供的核心价值可以概括为三层：

#### 第一层：内容筛选

帮助用户判断：

- 什么值得看
- 什么值得学
- 什么值得落地

#### 第二层：结构化组织

帮助用户在“文章很多”的情况下仍然能快速建立路径感，而不是被时间流淹没。

#### 第三层：长期复用

站点的终极价值不只是阅读，而是把知识变成：

- 可搜索
- 可引用
- 可复习
- 可迁移到真实工作流中的内容资产

## 5. 产品结构设计

### 5.1 整体产品结构

当前产品推荐的核心结构如下：

```text
首页
├─ 首屏定位
├─ 核心专题入口
│  ├─ AI Agent
│  ├─ Coding Agent
│  └─ 开源 AI 工具
├─ 深度内容入口
│  ├─ 技术笔记
│  └─ 视频精读
├─ 时效内容入口
│  └─ 行业快讯
└─ 辅助入口
   ├─ 搜索
   ├─ About
   └─ Contact

专题页
├─ 适合谁看
├─ 先看哪几篇
├─ 读完会得到什么
├─ 系统阅读路径
├─ 精选文章
├─ 相关专题
└─ 轻转化入口

文章页
├─ 正文
├─ 相关文章
├─ 专题回流
├─ 搜索 / 标签入口
└─ 轻转化入口
```

### 5.2 核心页面层级

#### 一级页面

- 首页 `/`
- About `/about/`
- Contact `/contact/`
- Search `/search/`

#### 二级页面

- 核心专题页
- 分类页
- 标签页

#### 三级页面

- 单篇文章页

### 5.3 内容层级原则

内容组织不应再完全依赖“最新发布顺序”，而应按用户目标重组：

- 首页负责第一次进入时的分流与理解
- 专题页负责系统学习和目标导向阅读
- 文章页负责深度消费和继续探索
- 搜索 / 分类 / 标签页负责问题导向的查找

### 5.4 专题扩张规则

新专题不应因为“有几篇相关文章”就创建，而应同时满足以下条件中的至少 3 条：

- 该主题已有 5 篇以上可归入的文章
- 该主题能覆盖一个稳定用户问题，而不是临时热点
- 该主题能形成从入门、判断到实践的阅读路径
- 该主题与现有专题存在明确互链关系
- 该主题有后续 4 周以上的内容维护计划

专题页上线后，每月至少检查一次：

- 是否有过期文章需要降权
- 是否有新文章应加入精选或路径
- 是否需要增加跨专题链接
- 是否存在与其他专题合并的必要

## 6. 页面模块规格

### 6.1 首页模块规格

首页不是“展示所有内容”，而是承担定位、分流、品牌强化和轻转化 4 个职责。

| 顺序 | 模块 | 作用 | 内容要求 | 主动作 |
| --- | --- | --- | --- | --- |
| 1 | 首屏定位 | 解释站点是什么 | 一句话定位、三类主题关键词、搜索入口 | 进入专题或搜索 |
| 2 | 核心专题入口 | 承担主导航 | 3 个专题卡片，每个卡片包含适合谁看、先读哪篇 | 进入专题页 |
| 3 | 深度内容精选 | 展示长期价值 | 技术笔记、视频精读、高价值文章 | 阅读文章 |
| 4 | 时效内容 | 保留更新感 | 行业快讯独立展示，不抢占首屏 | 查看快讯 |
| 5 | 辅助入口 | 建立信任与联系 | About、Contact、分类、标签 | 了解站点或联系 |

首页设计约束：

- 首屏必须出现 `AI Agent`、`Coding Agent`、`开源 AI 工具` 这 3 个主题信号
- 专题入口优先级高于最新文章列表
- 快讯不能成为首页主视觉或首要分流入口
- 移动端首屏应至少露出一个明确专题入口

### 6.2 专题页模块规格

专题页是当前站点最关键的产品资产，应从导览页升级为专题中心。

| 顺序 | 模块 | 作用 | 内容要求 | 主动作 |
| --- | --- | --- | --- | --- |
| 1 | 专题定义 | 解释该专题解决什么问题 | 一句话说明、适合用户、主题边界 | 判断是否继续 |
| 2 | 起步阅读 | 降低进入成本 | 2 到 4 篇“先读文章” | 进入第一篇文章 |
| 3 | 阅读路径 | 支持系统学习 | 入门、进阶、实践或工具选择路径 | 按路径阅读 |
| 4 | 精选文章 | 承接深度阅读 | 高价值文章卡片，优先长期内容 | 阅读文章 |
| 5 | 相关专题 | 建立知识网络 | 与其他专题互链，并解释关系 | 跳转专题 |
| 6 | 轻转化 | 承接认可用户 | Contact、About、后续订阅入口 | 建立关系 |

专题页设计约束：

- 进入专题页后，用户应在 10 到 20 秒内知道从哪一篇开始读
- 每个专题至少提供 1 条主阅读路径
- 每个专题至少提供 2 个跨页面去向：文章页和相关专题页
- 专题页 CTA 不应只写“了解更多”，应明确下一步动作

### 6.3 文章页模块规格

文章页是内容消费主阵地，但不应成为阅读终点。

| 位置 | 模块 | 作用 | 内容要求 |
| --- | --- | --- | --- |
| 正文前 | 主题归属 | 让用户知道文章属于哪条线 | 专题、分类、标签 |
| 正文中 | 相关概念入口 | 在合适位置补充跳转 | 只放强相关链接，避免干扰阅读 |
| 正文后 | 继续阅读 | 引导下一篇 | 2 到 4 篇相关文章 |
| 正文后 | 专题回流 | 引导系统学习 | 回到所属专题或相关专题 |
| 页面底部 | 轻转化入口 | 建立关系 | Contact、About 或后续订阅入口 |

文章页设计约束：

- 底部 CTA 不能只停留在“返回首页”
- 相关文章应优先同专题、同问题、同工具链，不只按发布时间推荐
- 文章页应能承接外部流量，让用户发现站点的主题结构

### 6.4 搜索页模块规格

搜索页不是附属功能页，而是问题导向用户的主入口之一。

搜索页应提供：

- 搜索输入框
- 常见主题提示
- 核心专题入口
- 无结果状态下的引导入口

无结果状态不应只显示“没有找到结果”，而应引导用户尝试：

- 更短关键词
- 核心专题页
- 分类页或标签页
- Contact 反馈内容需求

### 6.5 分类页与标签页模块规格

分类页负责较稳定的内容组织，标签页负责更灵活的问题聚合。

分类页和标签页应提供：

- 当前分类或标签的简短解释
- 推荐先读内容
- 最新内容列表
- 相关分类、标签或专题入口

## 7. 用户路径设计

### 7.1 路径一：首页 -> 专题页 -> 文章页

适用于首次进入的新用户。

目标：

- 先理解站点
- 再选择主题
- 再进入具体内容

设计要求：

- 首页专题入口足够明确
- 专题页能迅速给出“从哪里开始”
- 文章页能继续回流到专题体系

### 7.2 路径二：搜索 / 外部流量 -> 文章页 -> 专题页

适用于从搜索引擎、社交平台、收藏链接进入的用户。

目标：

- 单篇文章先完成价值交付
- 再引导用户发现更大的内容结构

设计要求：

- 文章页必须有清晰的专题推荐与继续阅读
- 文章页底部 CTA 不能只停留在“返回首页”
- 文章页应解释该内容所属主题，而不是只展示单篇正文

### 7.3 路径三：首页 -> 搜索 / 分类页 -> 文章页

适用于已有方向但没有精确主题的用户。

目标：

- 通过栏目或搜索缩小范围
- 再进入单篇或专题

设计要求：

- 首页应有可见搜索入口
- 分类页不应是纯归档，应提供主题解释和推荐内容
- 搜索页应能在无结果时给出替代路径

### 7.4 路径四：专题页 -> About / Contact

适用于愿意进一步建立关系的用户。

目标：

- 从内容消费进入品牌理解
- 从品牌理解进入转化或联系

设计要求：

- Contact 入口不应干扰阅读路径
- About 应解释站点为什么值得长期关注
- 后续订阅能力上线前，Contact 是主要中转入口

## 8. 转化设计

### 8.1 当前转化阶段判断

当前更适合做 `轻转化 + 中转化`，不建议过早做过强的商业动作。

原因：

- 站点主价值仍然是内容结构和主题权威性
- 用户更需要建立信任，而不是立刻被要求订阅或购买
- 过早强化商业入口会稀释“知识入口”的首要心智

### 8.2 转化层级

| 层级 | 用户动作 | 当前状态 | 设计重点 |
| --- | --- | --- | --- |
| 轻转化 | 进入专题、继续阅读、使用搜索、浏览分类 | 当前优先 | 降低下一步成本 |
| 中转化 | 查看 About、Contact、建立进一步连接 | 当前保留 | 保持克制但可见 |
| 强转化 | 邮件订阅、社群、咨询合作、产品化服务 | 后续扩展 | 等内容结构成熟后推进 |

### 8.3 当前推荐策略

#### 首页

- 突出专题入口
- 保留 Contact / About / Search 作为轻量行动入口
- 不在首屏强推商业合作

#### 专题页

- 持续强化“下一步应该去哪”
- 增加专题之间的跳转矩阵
- 让专题页承担更多“持续关注某主题”的语义

#### 文章页

- 强化继续阅读
- 强化专题回流
- 保留联系入口，但不要打断正文阅读

## 9. 信息架构与内容分发原则

### 9.1 一级原则

- 先按用户目标组织，再按发布时间组织
- 先按主题建立路径，再按单篇沉淀内容
- 先做结构清晰，再做内容扩张
- 先证明用户能走通路径，再追求更强转化

### 9.2 内容分发原则

#### 技术笔记

- 主承担长期搜索与知识沉淀
- 应获得首页和专题页更高权重

#### 视频精读

- 承担高价值外部内容的结构化转译
- 适合作为理解新主题的过渡入口

#### 行业快讯

- 承担覆盖与触达
- 不应长期挤占首页主心智
- 更适合独立区块或栏目承接

#### 辅助栏目

- 用于品牌表达和边缘主题补充
- 不承担主增长职责

## 10. 当前产品问题

### 10.1 已被缓解的问题

- 首页过去过于时间流导向，已初步改善
- 专题内容缺少正式入口，已建立 3 个核心专题页
- 文章页阅读终点过强，已加入继续阅读与转化入口
- 发布流程容易缺页，已加入校验

### 10.2 仍待解决的问题

#### 1. 首页与专题页联动仍然偏浅

虽然已有专题入口，但首页尚未完全形成“专题即核心导航”的结构。

优先动作：

- 将专题卡片前置到首页核心区域
- 为每个专题补充人群标签和“先读哪篇”
- 在首页减少快讯对主路径的抢占

#### 2. 专题页离成熟专题中心还有距离

后续还需要继续沉淀：

- 更强的精选文章卡片
- 更明确的按目标阅读路径
- 更清晰的专题互链
- 更稳定的专题维护机制

#### 3. 订阅与沉淀路径不够明确

当前已有联系入口，但还没有更明确的“持续跟进这个主题”的产品语义。

优先动作：

- 在专题页和文章页保留轻量联系入口
- 后续订阅能力上线前，不把订阅作为主要 CTA
- 为未来订阅入口预留文案和组件位置

#### 4. 缺少面向产品的指标定义

目前站点更像内容运营系统，尚未形成一套更明确的产品指标框架。

优先动作：

- 先定义指标口径
- 上线后用 2 周建立基线
- 再基于基线做专题入口和文章回流优化

## 11. 指标设计

### 11.1 指标使用原则

当前不应用复杂指标系统压过产品建设。指标的作用是验证 3 件事：

- 用户是否理解站点定位
- 用户是否进入专题路径
- 用户是否从单篇阅读进入持续阅读

如果当前没有历史基线，先用上线后 2 周数据建立基线，再用后续 4 周观察趋势。

### 11.2 核心产品指标

| 指标 | 口径 | 观察周期 | 用途 | 当前阶段目标 |
| --- | --- | --- | --- | --- |
| 首页到专题页点击率 | 从首页访问中点击任一核心专题页的会话数 / 首页会话数 | 周 | 判断专题入口是否有效 | 建立基线后持续提升 |
| 专题页到文章页点击率 | 从专题页点击文章的会话数 / 专题页会话数 | 周 | 判断专题页是否能导读 | 建立基线后持续提升 |
| 文章页到专题页回流率 | 从文章页点击所属或相关专题的会话数 / 文章页会话数 | 周 | 判断文章页是否能回流主题 | 建立基线后持续提升 |
| 单次会话浏览页数 | 总浏览页数 / 会话数 | 周 | 判断站内探索深度 | 避免只读单篇即离开 |
| 搜索页使用率 | 搜索页会话数 / 总会话数 | 周 | 判断问题导向入口是否被使用 | 建立基线 |
| About / Contact 访问率 | About 与 Contact 会话数 / 总会话数 | 月 | 判断信任和联系入口是否可见 | 保持自然增长 |

### 11.3 内容指标

| 指标 | 口径 | 用途 |
| --- | --- | --- |
| 技术笔记搜索流量占比 | 技术笔记自然搜索会话数 / 全站自然搜索会话数 | 判断长期内容沉淀能力 |
| 快讯入口流量占比 | 快讯相关页面会话数 / 全站会话数 | 避免快讯稀释主品牌 |
| 专题页承接内容分布 | 每个专题关联文章数、精选文章数、路径文章数 | 判断专题是否健康 |
| 高价值文章长期访问曲线 | 核心文章按周访问趋势 | 判断内容是否具备复用价值 |

### 11.4 最小埋点建议

如果后续接入统计工具，至少记录以下事件：

- `topic_article_click`
- `article_topic_click`
- `article_related_click`
- `site_search_submit`
- `about_click`
- `contact_click`

每个事件至少包含：

- 来源页面

#### 12.2.3 首页实现拆解

首页优先处理 4 个模块，顺序如下：

1. 首屏定位区：保留当前站点定位，但把“先帮你判断什么值得看，什么值得学，什么值得落地”收束到 AI Agent、Coding Agent、开源 AI 工具三条线。
2. 核心专题卡片区：新增或改造 `home-topic-grid`，每张卡片包含：
   - 专题名
   - 适合谁看
   - 先读哪篇
   - 进入专题 CTA
3. 深度内容区：保留技术笔记和视频精读，但不要早于专题入口。
4. 快讯区：保留最近行业快讯，位置应晚于专题和深度内容。

首页卡片建议数据先写在模板内的 Hugo `dict` 中，等专题体系继续扩大后再迁移到站点参数或数据文件。

#### 12.2.4 专题页实现拆解

专题页需要把现在的 `audience`、`outcomes`、`starterPack` 扩展为更完整的数据契约：

```toml
[[readingPaths]]
title = "想先选工具"
description = "适合已经知道方向，但需要横向比较工具的读者。"
articles = [
  { title = "文章标题", url = "/posts/example/", note = "为什么先读这篇" }
]

[[featuredArticles]]
title = "文章标题"
url = "/posts/example/"
label = "精选文章"
note = "这篇文章在专题中的价值"

[[relatedTopics]]
title = "Coding Agent 工作流"
url = "/coding-agent/"
note = "把 Agent 能力落到真实工程流程"

[topicCta]
title = "继续沿着这个主题跟进"
description = "反馈选题、推荐工具或讨论合作。"
primaryText = "联系"
primaryUrl = "/contact/"
secondaryText = "站内搜索"
secondaryUrl = "/search/?q=AI%20Agent"
```

模板层优先读取结构化字段；正文 Markdown 继续保留，用来承载更长的解释和判断标准。这样专题页既能被人工阅读，也能稳定生成导读模块。

#### 12.2.5 文章页实现拆解

文章页 P1 的目标不是增加更多装饰，而是让外部流量不会停在单篇文章。

实现顺序：

1. 在文章正文前或元信息下增加“所属主题”区域。
2. 若文章 Frontmatter 有 `topics` 字段，优先按 `topics` 映射专题页。
3. 若没有 `topics` 字段，先按标签关键词回退匹配：
   - `AI Agent`、`Agent`、`MCP` -> `/ai-agent/`
   - `Claude Code`、`Codex`、`Cline`、`Coding Agent` -> `/coding-agent/`
   - `开源工具`、`工具`、`框架`、`Workflow` -> `/open-source-ai-tools/`
4. 相关文章推荐优先级调整为：同 `topics` > 同标签 > 同分类 > 当前 section 相邻文章。
5. 文章底部 CTA 保持轻量，不把商业合作放在第一动作。

如果新增 `topics` 字段，建议采用单行数组：

```toml
topics = ["coding-agent"]
```

对应关系：

| `topics` 值 | 专题页 |
| --- | --- |
| `ai-agent` | `/ai-agent/` |
| `coding-agent` | `/coding-agent/` |
| `open-source-ai-tools` | `/open-source-ai-tools/` |

#### 12.2.6 搜索页与分类页实现拆解

搜索页的 P1 重点是“无结果也不死路”：

- 保留当前 Pagefind 搜索 UI
- 扩展关键词建议，把 `AI Agent`、`Coding Agent`、`Claude Code`、`MCP`、`开源 AI 工具` 作为优先建议
- 在搜索区域下方保留 3 个专题入口
- 用 JavaScript 监听 Pagefind 无结果状态，展示替代路径区

分类页和标签页的 P1 重点是“聚合页有解释”：

- 分类页说明内容载体，例如技术笔记、视频精读、行业快讯
- 标签页说明主题连接，例如具体工具、概念、框架
- 页面顶部保留搜索入口
- 对 AI Agent、Coding Agent、开源工具相关标签，优先露出对应专题页

如果本轮改动涉及搜索页或 Pagefind 行为，再执行：

## 13. 验收标准

### 13.1 产品层面验收

一份“产品层面可接受”的版本，至少应满足：

- 用户首次进入首页时能快速理解站点定位
- 用户能在首页快速判断自己该进入哪一个专题
- 用户进入专题页后能在 10 到 20 秒内知道从哪开始读
- 用户读完文章后不会断路，而是能自然进入专题页或相关文章
- 搜索页、分类页、标签页不是纯归档，而能继续导流
- 站点发布时，首页、专题页、搜索页与关键入口不会静默缺失

### 13.2 页面级验收

| 页面 | 必须满足 |
| --- | --- |
| 首页 | 首屏定位清楚；核心专题可见；搜索入口可见；快讯不抢占主路径 |
| 专题页 | 有适合人群、先读文章、阅读路径、精选文章、相关专题 |
| 文章页 | 有所属专题、继续阅读、专题回流、轻转化入口 |
| 搜索页 | 有搜索框、无结果引导、核心专题入口 |
| 分类 / 标签页 | 有主题解释、推荐内容、相关入口 |

### 13.3 内容维护验收

- 新文章发布时，应判断是否归入已有专题
- 核心专题每月至少维护一次精选文章和阅读路径
- 快讯内容不应长期占据首页核心推荐位
- 高价值技术文章应优先进入专题页和相关文章推荐

## 14. 风险与边界

### 14.1 主要风险与缓解动作

| 风险 | 表现 | 缓解动作 |
| --- | --- | --- |
| 继续扩内容而不扩结构 | 文章增加，但用户找不到路径 | 新文章必须判断专题归属；每月维护专题页 |
| 快讯占比过高 | 首页主心智变成资讯流 | 快讯放入独立区块；首页优先展示专题和长期内容 |
| 转化动作过早过强 | 用户尚未建立信任就被要求订阅或合作 | 当前以轻转化为主；商业入口只保留在 About / Contact |
| 专题过度扩张 | 专题数量增加但质量下降 | 使用专题准入规则；不满足条件的内容先归入标签或分类 |
| 指标过度复杂 | 花太多时间做统计系统 | 先落地最小事件集，建立基线后再扩展 |

### 14.2 当前边界

本文档主要聚焦：

- 站点内容产品层
- 信息架构与页面职责
- 用户路径与转化设计
- 最小可执行指标口径

不覆盖：

- 编辑团队流程细则
- 完整 SEO 运营方案
- 商业化方案细则
- 数据埋点实施技术细节
- 视觉设计稿和组件实现代码

## 15. 结论

Text Matrix 当前最正确的产品方向，不是继续把自己做成“更大的内容站”，而是继续把自己做成“更清晰的主题入口”。

后续所有重要改进，都应服务于同一个目标：

- 让用户更快理解站点
- 让用户更快进入正确专题
- 让用户更容易沿着主题持续阅读
- 让内容从一次性消费，逐步变成长期可复用的知识资产

V1.1 的重点，是把这个方向变成可执行的页面结构、指标口径和路线图。只要首页、专题页和文章页能稳定形成闭环，Text Matrix 的差异化就不会来自“文章数量更多”，而会来自“它比别的站更会组织 AI Agent 与开发工作流内容”。
