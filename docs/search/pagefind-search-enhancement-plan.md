> 结论先行
>
> 这次搜索增强不适合“一次性做完”。更稳妥的做法是拆成三个独立阶段：
>
> 1. 结果卡片增强：先显示日期、分类、标签。
> 2. 筛选增强：再补年份、内容类型、分类筛选。
> 3. 搜索体验优化：最后再考虑排序、结果卡片样式、筛选联动与统计文案。
>
> 对 text-matrix 这类 Hugo + LoveIt + Pagefind 的静态站，最合理的工程策略不是“功能一次堆满”，而是先保证结果纯度，再逐步增加结果可读性与筛选能力。

## 1. 背景

当前仓库已经完成了两件关键工作：

1. 站内搜索已从主题内置搜索路线切到 Pagefind。
2. 已通过 `data-pagefind-body` 与 `data-pagefind-ignore` 缩小索引范围，只索引正文主体。

这意味着当前搜索已经“能用”，但还没有做到“信息充分”和“可筛选”。具体表现为：

- 搜索结果卡片主要显示标题和摘要。
- 用户在结果列表中不能直接看到发布日期、分类、标签。
- 站内内容量继续增长后，缺少按年份或内容类型缩小结果集的能力。

因此后续增强的目标不是重做搜索，而是在现有 Pagefind 架构上继续补两层能力：

- 结果卡片的元信息展示。
- 可控且低复杂度的筛选能力。

## 2. 目标与非目标

### 2.1 目标

本计划要解决以下问题：

1. 在搜索结果中直观看到日期、分类、标签。
2. 支持按年份、内容类型筛选结果。
3. 让改动保持最小，不破坏现有 Hugo 构建和 Pagefind 构建链路。
4. 保持 GitHub Pages 与 Cloudflare Pages 的双部署兼容。

### 2.2 非目标

本轮不做以下事情：

1. 不重写整个搜索 UI 组件。
2. 不替换 Pagefind UI 为自研前端搜索框架。
3. 不引入复杂的多条件高级查询语法。
4. 不在本轮接入服务端搜索或外部搜索服务。
5. 不在本轮做搜索日志分析、点击埋点和搜索推荐。

## 3. 现状评估

当前搜索相关结构可以分成四层：

| 层级 | 当前状态 | 后续改动重点 |
| ---- | ---- | ---- |
| 索引范围 | 已完成正文主体索引 | 保持不动 |
| 页面元信息 | 已有基础 `data-pagefind-meta`，但未完整用于 UI | 继续补充结构化字段 |
| 搜索页面 | 已有独立 `search` 页面和 Pagefind UI 初始化 | 增强结果展示和筛选配置 |
| 样式层 | 已有搜索页基础样式 | 只补结果 meta 与筛选面板样式 |

当前实现的优点：

- 搜索入口已稳定。
- 搜索页独立，适合逐步增强。
- Pagefind 构建链路已经跑通。
- 公共壳层与 archive 页面已基本排除在正文索引之外。

当前实现的不足：

- 结果卡片信息密度不够。
- Pagefind 元数据和前端展示之间还没有打通。
- 筛选字段尚未进入正式设计和验证阶段。

## 4. 设计原则

### 4.1 分阶段交付

不要把“结果展示增强”和“筛选增强”绑成一次提交。

原因：

- 结果卡片增强主要是前端展示问题。
- 筛选增强会牵涉 `data-pagefind-filter`、Pagefind UI filters、索引验证。
- 两者一起做时，排错成本会明显上升。

### 4.2 先改模板与元数据，再改搜索页 UI

Pagefind 的结果展示建立在“页面本身已经输出了稳定元信息”的前提上。

因此顺序应是：

1. 页面模板输出稳定的 meta 与 filter 字段。
2. 搜索页消费这些字段并决定如何展示。

### 4.3 保留 Pagefind UI 默认结构，避免过早自定义过深

当前最稳妥的路线不是重写搜索结果模板，而是：

- 继续使用 Pagefind UI。
- 通过 `processResult` 做轻量结果增强。
- 通过 CSS 对默认 DOM 结构做定制。

这样做的好处是：

- 改动范围小。
- 升级 Pagefind 时更稳。
- 不需要自己维护完整的分页、结果态、空态、筛选状态机。

### 4.4 筛选字段只保留高价值、低歧义字段

首批筛选字段建议只做：

1. 年份
2. 内容类型
3. 分类

不建议首批就做标签筛选，原因是：

- 当前站点标签数量多，长尾分散。
- 标签筛选面板很容易过长。
- 用户体验上更适合先把标签作为结果卡片信息展示，而不是主筛选维度。

## 5. 目标方案

### 5.1 结果卡片目标形态

每条搜索结果建议包含以下信息：

- 标题
- 摘要
- 日期
- 内容类型
- 分类
- 标签
- 子结果（如果命中具体章节）

推荐视觉层级：

1. 第一层：标题
2. 第二层：日期 + 内容类型
3. 第三层：分类 + 标签
4. 第四层：摘要

原因：

- 日期帮助用户判断内容时效性。
- 内容类型帮助区分“文章”和“文档”。
- 分类帮助快速定位内容语境。
- 标签用于补足主题线索，但不应压过标题和摘要。

### 5.2 筛选目标形态

首批筛选建议采用左侧或上方折叠筛选面板，包含：

- 类型：文章 / 文档 / 页面
- 年份：按实际年份离散值展示
- 分类：展示高频分类

交互原则：

- 默认收起分类，默认展开类型和年份。
- 空结果筛选项隐藏。
- 手机端筛选面板放到结果区上方。

### 5.3 元数据与筛选字段设计

建议统一输出如下字段：

| 字段 | 类型 | 用途 |
| ---- | ---- | ---- |
| `title` | meta | 结果标题 |
| `date` | meta | 原始可排序日期 |
| `date_display` | meta | 搜索结果里展示的日期文本 |
| `year` | meta | 展示与调试辅助 |
| `content_type` | meta | 结果卡片展示“文章 / 文档 / 页面” |
| `category` | meta | 结果卡片显示分类 |
| `tags` | meta | 结果卡片显示标签 |
| `类型` | filter | 筛选内容类型 |
| `年份` | filter | 筛选年份 |
| `分类` | filter | 筛选分类 |

说明：

- `meta` 用于展示。
- `filter` 用于筛选。
- 不要把“展示字段”和“筛选字段”混在一个概念里，否则后续排查困难。

## 6. 实施分期

### 6.1 阶段一：结果卡片增强

目标：

- 只让结果卡片显示日期、分类、标签。
- 不引入筛选。

涉及文件：

- `layouts/posts/single.html`
- `layouts/_default/single.html`
- `layouts/search/single.html`
- `assets/css/_custom.scss`

实施内容：

1. 在正文模板里输出 `date_display`、`category`、`tags` 等 Pagefind meta。
2. 在搜索页里通过 `processResult` 读取 `result.meta`。
3. 将 meta 注入摘要上方或标题下方。
4. 在样式层补结果卡片的 meta 行和 pill 样式。

验收标准：

- 搜索结果卡片能稳定显示日期。
- 有分类的页面显示分类。
- 有标签的页面显示标签。
- 不影响现有搜索输入、分页、加载更多和子结果展示。

复杂度评估：低到中。

### 6.2 阶段二：筛选增强

目标：

- 支持按类型、年份、分类筛选。

涉及文件：

- `layouts/posts/single.html`
- `layouts/_default/single.html`
- `layouts/search/single.html`
- `assets/css/_custom.scss`

实施内容：

1. 在正文模板中增加 `data-pagefind-filter` 字段。
2. 让搜索页初始化时开启 filters 配置。
3. 设置 `showEmptyFilters: false`。
4. 通过 `openFilters` 控制默认展开的筛选组。
5. 为筛选面板补桌面与移动端样式。

验收标准：

- Pagefind 构建日志能识别非零 filters。
- 搜索页出现“类型 / 年份 / 分类”筛选面板。
- 勾选筛选项后结果数量随之变化。
- 手机端布局不挤压搜索输入框和结果列表。

复杂度评估：中。

### 6.3 阶段三：体验修整

目标：

- 统一搜索页视觉与信息节奏。
- 解决筛选数量过多时的视觉噪音。

实施内容：

1. 优化结果卡片内元信息的间距与换行策略。
2. 优化筛选面板吸顶、折叠与移动端布局。
3. 如有必要，再补结果统计文案与筛选激活态样式。

验收标准：

- 搜索页在桌面与移动端都没有明显布局问题。
- 结果信息不会挤占摘要主体。
- 筛选区域的视觉权重低于搜索结果本身。

复杂度评估：低。

## 7. 文件级改造说明

### 7.1 `layouts/posts/single.html`

职责：文章页正文模板。

计划改动：

- 生成稳定的日期字段。
- 输出 `date_display`、`year`、`content_type` 等 meta。
- 输出 `类型`、`年份`、`分类` 这三个 filter。

注意事项：

- 日期应优先用 `PublishDate`，为空时回退 `Date`。
- `content_type` 对文章页固定为“文章”。
- 多个分类应逐个输出 filter，而不是拼成一个逗号串。

### 7.2 `layouts/_default/single.html`

职责：普通单页和 docs 页面正文模板。

计划改动：

- 与文章模板保持同一套 meta 结构。
- 根据 section 推导“文档 / 页面 / 文章”。

注意事项：

- docs 页与 about/contact/privacy 这类页面的内容类型要有明确区分。
- 避免因为模板分叉导致字段名不一致。

### 7.3 `layouts/search/single.html`

职责：搜索页主模板与 Pagefind UI 初始化入口。

计划改动：

- 阶段一：增加 `processResult`，把元信息拼到结果卡片里。
- 阶段二：增加 `showEmptyFilters`、`openFilters` 和筛选文案。

注意事项：

- 只做轻量增强，不改写整个搜索 UI 生命周期。
- 需要统一 HTML 转义，避免把 meta 文本直接拼成不安全 HTML。

### 7.4 `assets/css/_custom.scss`

职责：搜索页自定义样式。

计划改动：

- 阶段一：增加结果卡片 meta row、chip、pill 样式。
- 阶段二：增加筛选面板、checkbox、分组标题、移动端布局样式。

注意事项：

- 保持与当前搜索页 Hero 区和卡片风格一致。
- 不要让筛选面板比结果列表更抢眼。

## 8. 验证方案

### 8.1 本地验证

每一阶段完成后，至少执行：

```bash
npm run build:pagefind
```

重点观察：

1. Hugo 是否构建成功。
2. Pagefind 是否仍然识别 `data-pagefind-body`。
3. 阶段二时，Pagefind 日志里的 `Indexed X filters` 是否大于 0。

### 8.2 页面验证

需要手动验证以下场景：

1. 搜索有结果时，卡片信息是否完整。
2. 搜索无结果时，空态文案是否正常。
3. 搜索命中文档、文章、普通页面时，类型是否正确。
4. 手机端搜索页是否出现布局溢出。

### 8.3 部署验证

如果阶段二上线，需要在两个环境各验证一次：

1. GitHub Pages
2. Cloudflare Pages

重点确认：

- 子路径下 `pagefind/` 资源仍能正常加载。
- 搜索页筛选面板与结果卡片未因 `baseURL` 差异失效。

## 9. 风险与回滚策略

### 9.1 主要风险

| 风险 | 表现 | 应对方式 |
| ---- | ---- | ---- |
| meta 字段不一致 | 某些页面结果缺日期或类型 | 统一模板字段名 |
| filter 未被索引 | 页面有 `data-pagefind-filter`，但构建日志仍是 0 filters | 先检查 HTML 产物，再检查 Pagefind 支持方式 |
| 搜索页样式过重 | 筛选面板抢占结果区 | 分阶段上线，先结果展示后筛选 |
| 移动端布局拥挤 | 搜索输入、筛选面板和结果堆叠混乱 | 单独补移动端断点样式 |

### 9.2 回滚策略

如果某一阶段效果不好，应按最小单位回滚：

1. 阶段一只回滚 `processResult` 和结果样式。
2. 阶段二只回滚 filter 输出和 filters 配置。
3. 不要把搜索入口、索引范围优化、构建链路一起回滚。

原则是：

- 搜索功能可退化。
- 搜索基础能力不要被误伤。

## 10. 推荐执行顺序

后续真正恢复实施时，推荐按下面顺序推进：

1. 先做阶段一，只增强结果卡片。
2. 验收通过后，再做阶段二筛选。
3. 最后再做阶段三视觉与交互修整。

推荐原因：

- 阶段一最容易感知收益。
- 阶段一不依赖 Pagefind filter 行为验证。
- 阶段二一旦遇到 filter 采集异常，也不会阻塞结果卡片的改进价值。

## 11. 预计工作量

按保守估计：

| 阶段 | 预计工作量 | 风险等级 |
| ---- | ---- | ---- |
| 阶段一：结果卡片增强 | 0.5 天 | 低 |
| 阶段二：筛选增强 | 0.5 到 1 天 | 中 |
| 阶段三：体验修整 | 0.5 天 | 低 |

如果要压缩成一次短迭代，建议只做阶段一。

## 12. 最终建议

如果以后重新启动这项工作，建议不要再以“把搜索增强一次做完”为目标，而是改成如下方式：

1. 第一轮只做结果卡片增强。
2. 第二轮单独验证 Pagefind filters。
3. 第三轮再决定是否需要更深的搜索体验重构。

这样做的好处是：

- 风险更低。
- 每轮都有独立可见收益。
- 一旦中途暂停，也不会留下半套复杂实现。
