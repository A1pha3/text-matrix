国产 Coding Plan 已经不能只比月费。页面看起来都是"几十元起、接入 Claude Code"，实际卖的却不是同一种东西：方舟卖多模型入口，GLM 卖单一模型家族的高额编程配额，MiniMax 和 MiMo 把代码、多模态与自家模型放进同一套额度。DeepSeek 更接近按量 API，WorkBuddy 则是办公智能体。把它们按价格排成一列再挑最便宜的，第一步就走错了方向。

> **信息快照：2026 年 6 月 21 日。** 套餐价格、模型名单、限额和邀请活动都会调整，购买前请以官方页面为准。文中的部分入口带邀请码，可能触发新用户优惠或邀请奖励；这不影响下文的比较结论。

## 先给结论

| 你的工作方式 | 优先考虑 | 原因 |
| --- | --- | --- |
| 经常在 Doubao、GLM、DeepSeek、Kimi、MiniMax 之间切换 | 方舟 Coding Plan | 一个订阅覆盖多个模型，支持手动切换和 Auto 调度 |
| 已经确认 GLM 是主力模型，代码任务占绝大多数 | GLM Coding Plan | 额度和 MCP 权益都围绕 GLM 编程链路设计 |
| 编程之外还要生成图片、语音、音乐或视频 | MiniMax Token Plan | 文本、代码和媒体生成共用一套额度 |
| 想用 MiMo 模型，并需要全模态、语音或 MiMo Claw | Xiaomi MiMo Token Plan | 四档订阅覆盖 MiMo 模型、主流编程工具与可选 Claw 服务 |
| 用量很低，或要把模型接进线上服务 | DeepSeek API 等按量方案 | 按 Token 付费，低频时不用承担月费，也更符合生产调用方式 |
| 主要交付文档、PPT、数据分析，并依赖腾讯办公生态 | WorkBuddy | 它解决的是办公任务编排，不是单纯的 Coding 配额 |

还不确定自己属于哪一类的，先别买年费。连续记录 7 天的模型调用、峰值时段、长任务数量和多模态用量，答案通常会自己浮出来。

## 四套方案卖的不是同一种东西

主表里的四个方案看似都在"卖 Claude Code 入口"，但定位差异比价格差异更大。

| 方案 | 核心卖点 | 模型范围 | 能力范围 | 计量单位 |
| --- | --- | --- | --- | --- |
| 方舟 Coding Plan | 多模型入口 + Auto 调度 | Doubao / GLM / DeepSeek / Kimi / MiniMax 等 | 纯代码 | 请求数（估算） |
| GLM Coding Plan | 单模型家族深度配额 | GLM-5.2 / GLM-5-Turbo / GLM-4.7 | 代码 + 视觉/搜索/网页/仓库 MCP | prompt 次数 |
| MiniMax Token Plan | 代码与多模态共享额度 | MiniMax M3 / M2.7 + 图像/语音/音乐/视频 | 代码 + 全模态 | Token |
| Xiaomi MiMo Token Plan | MiMo 模型 + Claw 加购 | MiMo V2.5 系列 + 语音 | 代码 + 语音 + 可选 Claw | Credits |

四个方案在两个维度上拉开距离：**模型范围**（单模型家族 ↔ 多模型入口）和**能力范围**（纯代码 ↔ 多模态/办公）。方舟往"多模型"一侧走，GLM 往"单模型深度"一侧走，MiniMax 和 MiMo 往"多模态"一侧走。DeepSeek 和 WorkBuddy 不在这张表里，因为它们根本不是同一种产品——前者是按量 API，后者是办公智能体。

定位不同，额度口径就不同，价格也就不能直接除一下算"每元买多少次调用"。

## 比价格之前，先统一五个口径

### 1. 买到的是模型，还是模型入口

方舟的优势来自"可切换"：同一个套餐里有 Doubao、GLM、DeepSeek、Kimi、MiniMax 等模型。其余三家主要提供自有模型。前者降低切换成本，后者通常把配额、工具和附加能力围绕一个模型家族做得更集中。

经常在不同模型之间比较重构方案的人，方舟的入口价值很高；团队已经固定使用 GLM 的，继续为"模型自由"付费未必划算。

### 2. 额度单位能不能比较

不能。方舟按请求数给出估算，GLM 使用 prompt 次数，MiniMax 用 Token，MiMo 使用 Credits。同一个 Claude Code 任务可能触发多轮模型调用，长上下文、思考模式、Agent Team 和不同模型的倍率也会改变消耗。

"每 5 小时 1,200 次请求"和"每 5 小时 80 次 prompts"看起来前者大，实际上两者用不同的口径估算同一件事：方舟按特定模型和工具模式估"模型被调用多少次"，GLM 按"用户发起多少次 prompt"估，一次 prompt 预计会触发 15～20 次模型调用。把两个数字直接相减没有意义；记录自己的典型任务在每套方案里能完成多少次，才有比较的基础。

### 3. 限额是按月，还是还有短窗口

方舟、MiniMax 和 GLM 都公开说明了 5 小时或周窗口；MiMo 则有 Credits、时段倍率和不同档位的用量差异。再加上并发数与高峰期限流，实际使用往往会先碰到某个局部限制，而不是耗尽页面上最醒目的月度总量。白天同时跑多个 Agent 的用户尤其要看短窗口；偶尔做一次大重构的人更容易被单次长任务和周额度影响。

### 4. 套餐能不能用于生产 API

个人 Coding Plan 通常面向交互式编程工具，不等于生产 API 资源包。方舟明确说明，套餐额度只在支持的 Coding 工具中生效；GLM 也限定指定工具和产品环境；MiniMax 官方建议生产环境使用按量付费。

要给网站、机器人或 SaaS 后端提供稳定服务，应按 API 的价格、并发、服务条款和可用性重新评估，不能把个人订阅 Key 当成廉价生产通道。

### 5. 工具兼容不等于体验相同

"支持 Claude Code、Cursor、OpenCode"只说明能接入。模型是否支持图像、思考模式如何开启、工具调用是否稳定、上下文窗口怎样计费、接口兼容层是否完整，都会影响真实体验。选套餐前，至少用自己的仓库跑一次需求实现、一次跨文件重构和一次失败排查。

| --- | ---: | --- | --- | --- |
| 方舟 Coding Plan | 限时 9.9 元起 | Doubao、GLM、DeepSeek、Kimi、MiniMax 等 | Lite / Pro；5 小时、周、月三层限额；多工具共享额度 | 需要频繁切模型的个人开发者 |
| MiniMax Token Plan | Plus 49 元/月 | MiniMax M3、M2.7 及图像、语音、音乐模型 | Plus / Max / Ultra；5 小时和周窗口；高峰期可能动态限流 | 编程与内容生产混合的个人或小团队 |
| GLM Coding Plan | Lite 49 元/月 | GLM-5.2、GLM-5-Turbo、GLM-4.7 | Lite / Pro / Max；5 小时和周限额；高阶模型按倍率抵扣 | 把 GLM 作为主力编码模型的人 |

表里的价格只用于定位档位，不适合直接计算"谁最便宜"。方舟 9.9 元是限时首购入口，GLM 和 MiMo 页面同时展示原价、周期折扣或首购价，MiniMax 邀请活动也会改变实付金额。购买时应以续费价格和取消规则为准，首月折扣只算一次。

## 方舟 Coding Plan：多模型入口的账怎么算

方舟目前提供 Lite 和 Pro 两档。Lite 覆盖 Doubao、GLM、DeepSeek、Kimi、MiniMax 等模型，Pro 的用量约为 Lite 的 5 倍。用户可以在 Claude Code、Cursor、Cline、Kilo Code、OpenCode 等工具中使用同一套餐，额度在工具之间共享。

它适合两类人：一类经常比较不同模型的输出，另一类不想分别维护多家订阅和配置。Auto 模式还能减少手动选模型的次数，不过需要可复现结果时，最好固定模型，而不是把同一任务交给动态路由。

官方用 Doubao Seed 2.0 Lite、Claude Code 未开启 Agent Team 的场景做过估算：Lite 每 5 小时最多约 1,200 次请求、每周约 9,000 次、每订阅月约 18,000 次；Pro 为其 5 倍。这个数字会被模型、上下文长度和任务复杂度显著改变，不能当成承诺值。它测的是"在特定模型 + 特定工具模式下，5 小时窗口内能发起多少次模型调用"，换成 GLM-5.2 或开启 Agent Team，数字会立刻往下走；它也不能推出"方舟比 GLM 多 15 倍额度"，因为两边的单位根本不同。

需要留意两个边界：套餐额度不能用于普通 API 调用；额度耗尽后不会继续扣账户余额，只能等待窗口恢复。团队协作和线上服务应改用按量 API。

## MiniMax Token Plan：代码和多模态共用一套额度

MiniMax 当前同时服务编程与多模态任务。个人开发者既可以在 MiniMax Code 里直接使用 Token Plan，也可以通过订阅 Key 接入 Claude Code、Codex CLI、OpenCode、Cursor 等工具。

三档月费分别为 Plus 49 元、Max 119 元、Ultra 469 元。它们都支持文本、图像、语音和音乐，Max 与 Ultra 额外提供每日视频生成额度；档位越高，可并行运行的 Agent 数量和月度 Token 越多。

这套方案最容易被忽略的是限流。额度受 5 小时固定窗口和周窗口控制，高峰时段还可能动态调整；未用完的额度不会结转。官方也明确把 Token Plan 定位为个人交互式使用，生产环境建议按量付费。

日常工作是"写代码 → 生成配图 → 做语音或音乐素材"的人，共享额度很顺手。纯编程用户则要先算清楚：多模态权益没有被使用时，它们只是套餐介绍里漂亮但闲置的一行字。

- [MiniMax Token Plan](https://platform.minimaxi.com/subscribe/token-plan?code=DRm14bJp5I&source=link)
- [MiniMax Token Plan 快速接入](https://platform.minimaxi.com/docs/token-plan/quickstart)

## GLM Coding Plan：围绕单一模型家族做深

GLM Coding Plan 提供 Lite、Pro、Max 三档，月付原价分别为 49 元、149 元和 469 元；季付、年付及活动价会变化。所有档位目前支持 GLM-5.2、GLM-5-Turbo 和 GLM-4.7，并可接入 Claude Code、OpenClaw、OpenCode、TRAE、CodeBuddy 等工具。

套餐还包含视觉理解、联网搜索、网页读取和开源仓库 MCP。Lite、Pro、Max 每月后三类 MCP 的合计调用上限分别为 100、1,000、4,000 次，视觉理解 MCP 与模型共享 5 小时资源池。

用量按 prompt 估算：Lite 每 5 小时最多约 80 次、每周约 400 次，Pro 和 Max 分别约为 5 倍和 20 倍。一次 prompt 预计会触发 15～20 次模型调用，高阶模型在不同时间段还可能按 2 倍或 3 倍系数抵扣，所以不能把 prompt 数理解成普通聊天轮数。这里的 80 次 prompt 测的是"用户在 Claude Code 里按下回车的次数"，不是模型调用的次数；它反映的是 GLM 在典型编程任务下的 prompt 容量，不能直接换算成"能写多少行代码"或"能改多少个文件"。

已经验证 GLM 在自己的仓库里表现稳定的人，这类纵向套餐比多模型入口更直接。尚未确定主力模型就先买高档年费，会把试错变成沉没成本。

- [GLM Coding Plan](https://www.bigmodel.cn/glm-coding?ic=QPFQHHZ7CL)
- [GLM Coding Plan 套餐概览](https://docs.bigmodel.cn/cn/coding-plan/overview)

## Xiaomi MiMo Token Plan：从 API 试用入口变成完整订阅

把 MiMo 只写成"10 元体验金"已经过时。当前 Xiaomi MiMo Token Plan 有 Lite、Standard、Pro、Max 四档，月付原价分别为 39 元、99 元、329 元和 659 元，首购与年付另有折扣。套餐覆盖 MiMo V2.5 系列、语音识别和语音合成模型，并支持 Claude Code、OpenCode、Kilo Code、OpenClaw 等工具。

MiMo 使用 Credits 计量，不同模型和时段的消耗倍率会改变实际可用量；北京时间 00:00～08:00 按 0.8 倍系数消耗。页面还提供 MiMo Claw 加购项，面向 7×24 小时任务、并行执行和金山办公服务。这部分与普通 Coding Plan 不是同一种权益，购买时应分开核算。

另一个必须留意的点是模型迁移：官方已公告 V2 系列在 2026 年 6 月陆续转向 V2.5，并计划于 6 月 30 日下线旧模型名。依赖固定模型名称、输出风格或接口行为的项目，应先完成兼容性测试。

## 一次跨文件重构在四套方案里怎么走

抽象对比看多了，还是想象不出"一次真实任务在每套方案里会怎样消耗"。把一个 2,000 行的 `payment_service.py` 拆成 `validator`、`gateway`、`retry` 三个模块并补上对应测试，在 Claude Code 里发起，预计需要 90 分钟连续编码——这个任务足够小，能在一轮窗口里跑完，又足够复杂，能触发模型切换、MCP 调用和倍率消耗。

**方舟 Coding Plan。** 任务开始前手动切到 GLM-5.2 做主重构，遇到正则边界条件时切到 Doubao 比较一版替代写法，最后用 DeepSeek 复查测试覆盖。90 分钟里大约发起 6～8 次切换，每次切换不消耗额外配额，但每次模型调用都计入 5 小时窗口。Lite 的 1,200 次/5 小时在这次任务里大概率够用；如果同时还在跑另一个 Agent 做文档，就要看窗口是否被吃满。Auto 模式在这里不太合适——重构需要可复现，把模型选择交给动态路由会让结果难以复盘。

**GLM Coding Plan。** 全程固定 GLM-5.2，不切换模型。90 分钟里大约发起 5～6 次 prompt，每次 prompt 触发 15～20 次模型调用，合计约 100 次调用。Lite 的 80 次 prompt/5 小时够用，但要注意：如果开启思考模式或高阶模型倍率，单次 prompt 的消耗会翻倍。视觉理解 MCP 在这次任务里用不上，但仓库 MCP 可以用来快速调出 `payment_service.py` 的历史变更，算额外收益。

**MiniMax Token Plan。** 主重构用 M3，中间需要生成一张支付流程图插进文档。代码和图片共享同一套 Token 额度，不需要切换账户或额外付费。但要注意 5 小时窗口：如果上午已经跑过一轮编码，下午的重构可能撞上窗口恢复时间。纯代码任务在这里没有额外优势，多模态额度用不上时，相当于为不需要的能力付了费。

**Xiaomi MiMo Token Plan。** 任务在 00:00～08:00 发起的话，Credits 消耗打 0.8 折，90 分钟的重构实际计费约 72 分钟。如果任务必须在工作时段跑，就没有这个优惠。MiMo Claw 在这次任务里用不上——它是面向 7×24 小时长任务和并行执行的加购项，单次重构不需要。V2 → V2.5 的迁移期里，最好先确认 Claude Code 配置里的模型名已经更新，否则任务跑到一半可能撞上下线时间点。

四套方案都能完成这次重构，差异落在具体成本上：方舟省的是"切模型"的维护成本，GLM 省的是"配额围绕单一模型算得更准"，MiniMax 适合"代码 + 配图"混合任务，MiMo 适合"能调工作时段"的人。

## DeepSeek 与 WorkBuddy 为什么不放进主表

### DeepSeek：适合按量调用，但不是 Coding Plan

DeepSeek 官方当前按输入、输出 Token 计费，并从账户余额或赠送余额中扣款。它有 Claude Code 等 Agent 集成文档，但这不等于存在名为"DeepSeek Coding Plan"的订阅套餐。

低频调用、需要精确核算单次成本，或者要把模型接入生产服务时，按量 API 反而更合适。代价是 Agent 长任务可能连续发起大量请求，若没有预算上限、缓存和调用监控，账单会比固定订阅更难预测。

- [DeepSeek 模型与 API 定价](https://api-docs.deepseek.com/quick_start/pricing)

### WorkBuddy：比较的是交付范围，不是 Token 配额

WorkBuddy 面向办公任务：方案、PPT、数据分析、软件生成和音视频制作都在一个工作台里完成，并能连接腾讯文档、会议、邮箱、微信、企业微信和 ima。它可以参与软件开发，但购买理由通常是办公任务覆盖和腾讯生态连接，不是模型 API 或 Coding 配额。

一天里大半时间在整理材料、做演示和处理协作信息的人，WorkBuddy 可能比再买一个编码模型更有用；核心工作是大型代码库中的持续修改的人，仍应优先比较前面的 Coding Plan。

截至快照日，官方邀请页写明：新用户注册可获得 2,000 积分，活动截至 6 月 30 日。页面引用的《中国办公智能体平台市场研究报告 2026》没有提供可独立核查的报告入口，因此不把"最受欢迎"作为购买依据。

## 用 7 天数据选套餐

不要从最高档开始。先用 API 余额、免费额度或月付最低档完成一轮测试，每天记下下面五项：

1. **真实任务数**：需求实现、重构、测试、排错各完成多少次，不记聊天轮数。
2. **最重的 5 小时窗口**：连续编码时是否触发限额、排队或降速。
3. **模型切换次数**：切换是为了质量，还是因为某个模型经常失败。
4. **非代码用量**：图片、语音、视频、搜索和 MCP 调用到底占多少。
5. **溢出成本**：套餐耗尽后，是等待恢复，还是必须转按量 API 继续工作。

一周后把账算成：

> **有效月成本 = 订阅费 + 溢出调用费 + 闲置权益成本 + 切换与维护成本**

四个组成部分分别对应四种典型浪费：订阅费浪费在"买高了档位"，溢出调用费浪费在"买低了档位"，闲置权益成本浪费在"买了不需要的能力"（比如纯编程用户买 MiniMax 却从不用多模态），切换与维护成本浪费在"买了多模型入口却只用一个模型"。把一周数据填进这四个格子里，哪一项在漏水就一目了然。

举几个具体的判断方向：方舟可能月费不是最低，但少维护四套入口；MiniMax 的多模态额度如果每天都用，就不是赠品；GLM 的 MCP 调用若替代了其他搜索和仓库工具，也应计入收益；MiMo 的低谷倍率只对能调整工作时段的人有意义。

## 购买前最后检查一次

- 先月付，再决定季付或年付；至少跑完一次完整迭代周期。
- 用自己的仓库测试，不用官方 Demo 代替真实任务。
- 同时记录短窗口、周窗口和并发限制，别只看"月度总量"。
- 确认套餐是否只允许个人使用、是否限定工具，以及额度耗尽后的行为。
- 检查续费原价、自动续费取消时间、退款规则和邀请优惠期限。
- 线上服务、团队共享或批处理任务优先评估按量 API，不要依赖个人订阅。

## 怎么把这套判断搬到自己身上

四套方案的细节会变，价格会调，模型会迭代，但判断顺序相对稳定：先确定自己需要多模型入口、单模型深度、多模态额度，还是按量 API；再用一周真实任务测出短窗口和溢出成本；最后才看价格。错过一次折扣损失有限，选错产品类型却会让月费和维护成本持续漏水。把"先定位、再测用量、最后比价"这套顺序记下来，下次再看到新的 Coding Plan 上线，也不用从头算一遍。
