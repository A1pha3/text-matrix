# Dexter：一站式 AI 研究代理平台

> **难度**：⭐⭐⭐ | **类型**：进阶分析 | **预计阅读时间**：45 分钟
> **目标读者**：AI 开发者、金融研究者、对自己搭建本地 Agent 系统有兴趣的用户

---

## 🎯 概述：Dexter 不只是一个 CLI

当你第一次看到 Dexter 的介绍时，你可能会想："又一个 CLI 聊天工具？"

不。Dexter 的野心比这大得多。

**Dexter 是一个运行在终端里的 AI 研究代理平台。** 它的目标不是生成「看起来像答案」的文本，而是把你的问题转成一组研究动作，再调用金融、网页、文件、技能、记忆等工具完成任务。

这意味着 Dexter 的价值不在于「会聊天」，而在于「会做研究」。

---

## 🏛️ 核心架构：六层协同设计

Dexter 采用六层架构，每一层各司其职，层与层之间通过清晰接口通信：

```
┌─────────────────────────────────────────────┐
│                   入口层                      │
│           src/index.tsx、gateway             │
├─────────────────────────────────────────────┤
│                   交互层                     │
│       CLI 终端 UI、输入历史、状态显示        │
├─────────────────────────────────────────────┤
│                   代理层                     │
│        规划、迭代、上下文管理               │
├─────────────────────────────────────────────┤
│                   模型层                     │
│          多提供商适配、调用、重试           │
├─────────────────────────────────────────────┤
│                   工具层                     │
│    金融、网页、文件、自动化、记忆、技能      │
├─────────────────────────────────────────────┤
│                扩展能力层                    │
│       技能、记忆、网关、定时任务            │
└─────────────────────────────────────────────┘
```

### 为什么分层设计很重要？

| 问题现象 | 问题本质 |
|----------|----------|
| 模型回答差 | 不一定是模型问题，可能是工具没装配成功 |
| 网关收不到消息 | 不一定是网关问题，可能是调度没触发 |
| 记忆不生效 | 不一定是记忆问题，可能是初始化失败或检索条件不匹配 |

理解架构的目的不是「背目录」，而是知道一个问题最可能属于哪一层。

## 🤖 模型层：多模型统一路由

### 支持的提供商

Dexter 支持 8 家模型提供商：

| 提供商 | 模型前缀 | API Key |
|--------|---------|---------|
| OpenAI | 无前缀，如 `gpt-5.4` | `OPENAI_API_KEY` |
| Anthropic | `claude-` | `ANTHROPIC_API_KEY` |
| Google | `gemini-` | `GOOGLE_API_KEY` |
| xAI | `grok-` | `XAI_API_KEY` |
| Moonshot | `kimi-` | `MOONSHOT_API_KEY` |
| DeepSeek | `deepseek-` | `DEEPSEEK_API_KEY` |
| OpenRouter | `openrouter:` | `OPENROUTER_API_KEY` |
| Ollama | `ollama:` | 不需要云端 Key |

### 前缀路由机制

Dexter 根据模型名字判断提供商，而不是先让用户选提供商再选模型。这种设计的优势：

- **统一直接**：一个配置项解决所有路由问题
- **便于扩展**：新增提供商只需在 `providers.ts` 注册
- **灵活切换**：用户可以在运行时通过 `/model` 命令切换模型

### Anthropic 的特殊优化

对 Anthropic 模型，Dexter 会显式把系统提示标记为 `cache_control: ephemeral`。这是因为系统提示包含大量内容（工具描述、技能元数据、记忆信息），缓存前缀能显著降低重复输入成本。

## 🔧 工具层：六类工具协同

Dexter 的工具可以分为 6 大类：

| 类别 | 工具 | 主要用途 |
|------|------|----------|
| **金融研究** | `get_financials`、`get_market_data`、`read_filings`、`stock_screener` | 财务、价格、新闻、申报、筛选 |
| **网页浏览** | `web_search`、`web_fetch`、`browser` | 搜索、抓取、JavaScript 渲染 |
| **文件系统** | `read_file`、`write_file`、`edit_file` | 本地文件读写 |
| **自动化** | `heartbeat`、`cron` | 持续检查和定时任务 |
| **记忆** | `memory_search`、`memory_get`、`memory_update` | 持久记忆与会话回溯 |
| **技能** | `skill` | 调用 SKILL.md 工作流 |

### 金融工具的精细分工

```
get_financials     → 财务、指标、估值、分析师预期
get_market_data     → 股价、日内涨跌、新闻、内幕交易
read_filings        → SEC 文件（10-K、10-Q、8-K）原文
stock_screener      → 按条件筛选股票
```

### 网页工具的选型原则

| 工具 | 使用场景 | 成本 |
|------|----------|------|
| `web_search` | 通用信息查找 | 低 |
| `web_fetch` | 抓取静态页面正文 | 中 |
| `browser` | JavaScript 渲染、单页应用 | 高 |

**关键原则**：不默认全用 `browser`。Dexter 的系统提示明确要求仅在必要时使用，因为它的成本更高、速度更慢、失败率更大。

### 工具装配的隐藏逻辑

Dexter 只把「当前真的能用」的工具告诉模型。这不是技术限制，而是设计选择。

**为什么？**

如果系统提示列出一个不能调用的工具，模型会产生错误计划。用户会看到「模型说要调用工具 X」，然后工具 X 实际不可用——这种体验比「压根不展示这个工具」更差。

## 🧠 技能系统：可复用的工作流引擎

### 为什么需要技能？

并不是所有复杂任务都适合每次从零规划。例如：

- DCF 估值流程
- 系统化研究模板
- 固定格式产出

这类任务更适合用一套经过整理的工作流来指导代理，而不是每次都重新让模型猜。

### 技能系统的工作原理

```
1. 发现   → 扫描包含 SKILL.md 的目录
2. 暴露   → 仅注入技能名称和描述到系统提示
3. 按需   → Agent 匹配后调用 skill 工具
4. 执行   → 返回技能正文，解析相对路径
```

**为什么只暴露元数据，而不是全部正文？**

因为大多数查询根本不需要技能。如果把所有技能全文都塞进系统提示，只会增加上下文成本，而没有实际收益。

### 技能的原子性

每个技能只做一件事。这使得技能可以：

- 被单独测试
- 被组合使用
- 被版本化管理
- 被共享给其他 Agent

## 💾 记忆系统：跨越会话的知识积累

### 三种上下文，不要混为一谈

| 概念 | 作用域 | 用途 |
|------|--------|------|
| **Scratchpad** | 当前查询内 | 工具结果、思考过程 |
| **会话历史** | 当前 CLI 运行期间 | 多轮对话上下文 |
| **持久记忆** | 跨会话 | 用户偏好、长期信息 |

### 持久记忆的存储与检索

```
存储位置：.dexter/memory/
格式：Markdown 文件
检索：memory_search、memory_get、memory_update
```

### 混合检索机制

记忆系统不是单纯的全文匹配，而是结合：

- **向量检索**：语义相似度
- **文本检索**：关键词匹配
- **时间衰减**：优先返回新信息
- **MMR 去冗余**：避免重复上下文

### 为什么不用普通文件操作？

记忆不仅是文件写入，还涉及索引、检索语义和系统约定。绕过专用工具会破坏这一层抽象。

## 📱 WhatsApp 网关：从本地 CLI 到消息通道

### 三种使用模式

| 模式 | 特点 | 适用场景 |
|------|------|----------|
| **Self-chat** | 给自己发消息 | 个人快速使用 |
| **Bot phone** | 专用 Bot 号码 | 固定用户群 |
| **群聊** | @ 提及触发 | 团队协作 |

### 群聊的触发机制

群聊中必须通过 `@` 提及 Dexter 才能触发响应。

这不是技术限制，而是**必要的噪音控制**。如果机器人在群聊中对所有消息都响应，误触发会迅速失控。

### dmPolicy 和 groupPolicy

| 策略 | 含义 |
|------|------|
| `pairing` | 需要配对 |
| `allowlist` | 仅白名单可触发 |
| `open` | 开放 |
| `disabled` | 禁用 |

这些策略是真实的安全边界，不是 UI 装饰。

## ⏰ 自动化层：心跳与定时任务

### Heartbeat

Heartbeat 负责按周期检查一份监控清单。当监控对象满足条件时，主动通知用户。

**典型场景**：
- 股价突破阈值
- 财务指标异常
- 新闻面出现特定关键词

### Cron

Cron 负责通用的定时任务。与 Heartbeat 不同，Cron 更通用——触发时执行的是完整的 Agent 询问，而不是简单的条件检查。

**典型场景**：
- 每天早上生成市场摘要
- 定时抓取竞品信息
- 定期更新某个标的的分析

### 与网关的联动

Heartbeat 和 Cron 执行后，结果通常通过网关回传给用户。如果你的自动化任务已经建立，但通知始终看不到——**优先检查网关，而不是先怀疑调度本身失效**。

## 🔄 执行流程：一次查询的完整路径

### 完整链路

```
用户输入 → CLI 接收 → Agent 创建
    ↓
模型决定是否需要工具
    ↓
执行工具 → 结果写入 scratchpad
    ↓
上下文超限？→ 触发清理
    ↓
最终生成答案 → 返回给用户
```

### Scratchpad 的核心地位

Scratchpad 是 Dexter 当前查询内的「工作台」。所有工具结果、思考信息和工具调用记录都会进入这里。

**为什么它如此关键？**

代理系统的价值不只是会调用工具，而是能把多次工具结果累积起来形成后续推理依据。没有 scratchpad，多步研究就会退化成一组彼此孤立的 API 调用。

### 上下文控制机制

Dexter 不是无限上下文。当 token 超过阈值时：

1. 清理最早的工具结果
2. 在更高压力下触发 memory flush 流程

**目标不是「保留一切」，而是优先保留高价值上下文。**

## 🎓 系统提示：代理行为的核心驱动力

Dexter 的代理行为很大程度上来自系统提示。系统提示由多个部分组成：

- 基础身份和行为约束
- 工具描述（**描述即行为**）
- 技能元数据
- 记忆说明
- 心跳说明
- 渠道特定格式约束
- `SOUL.md` 身份文档

**关键洞察**：工具的「说明书」直接影响模型的行为。工具描述写得清楚，规划就更稳定；写得模糊，模型就更容易误用能力。

## 📊 与传统 CLI 的本质区别

| 维度 | 传统 CLI 聊天 | Dexter 研究代理 |
|------|-------------|----------------|
| **回答质量** | 依赖模型自身知识 | 依赖工具调用的真实数据 |
| **执行模式** | 单轮响应 | 多轮迭代、工具协同 |
| **记忆能力** | 无或仅限当次会话 | 跨会话持久记忆 |
| **自动化** | 无 | 心跳监控、定时任务 |
| **扩展性** | 固定能力 | 技能系统可扩展 |
| **通道** | 仅本地终端 | CLI + WhatsApp |

## 🚀 快速上手路径

### 环境准备

```bash
# 1. 安装依赖
bun install

# 2. 配置环境变量
cp env.example .env
# 至少填写：
# OPENAI_API_KEY=your-key
# FINANCIAL_DATUAL_DATASETS_API_KEY=your-key

# 3. 启动 CLI
bun run start
```

### 首次验证

建议用以下问题验证工具链是否工作：

```text
Compare Apple and Microsoft on revenue growth, operating margin, and free cash flow trend over the last 5 years.
```

**为什么不用「你是谁」？** 因为这类问题无法验证工具链是否工作正常。

## 📌 总结：Dexter 的定位

### 一句话定义

> **Dexter 是一个以研究为导向的 AI 代理平台，通过多模型路由、工具协同、技能复用、持久记忆和通道集成，让 AI 不是「会说」，而是「能查、能做、能监控、能进化」。**

### 核心价值矩阵

| 能力 | 给用户带来的价值 |
|------|-----------------|
| 金融工具链 | 专业级金融研究不需要手动查数据 |
| 多模型路由 | 根据任务选择最合适的模型 |
| 技能系统 | 复杂方法论可沉淀、可复用 |
| 持久记忆 | 长期协作不需要重复说明偏好 |
| WhatsApp 网关 | 随时随地发起研究 |
| 自动化监控 | 被动等待，主动通知 |

### 适合谁？

| 用户类型 | Dexter 能帮你做什么 |
|----------|---------------------|
| 金融分析师 | 自动化财报研究、行业对比 |
| 投资者 | 持续监控标的、系统化决策 |
| 研究者 | 快速抓取、整合多源信息 |
| 开发者 | 学习 Agent 系统架构、技能扩展 |
| 技术管理者 | 理解 AI Agent 的能力边界和设计取舍 |
