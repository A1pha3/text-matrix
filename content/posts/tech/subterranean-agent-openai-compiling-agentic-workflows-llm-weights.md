# 把工作流编译进模型权重：OpenAI「地下代理」论文深度解读

> **一句话核心判断**：表面编排（Orchestration）不是唯一选择——将工作流的流程逻辑编译进小模型的权重，反而能让 3B–8B 模型在 procedural 任务上以 **128–462 倍的成本优势**，达到前沿模型 87%–98% 的质量。重新训练一次只需要 30–50 分钟，不是笨重的离线批处理，而是 CI/CD 流水线里的常规操作。

---

## 这篇论文在说什么

### 背景：编排框架社区热度极高，但有三个被忽视的代价

2024–2026 年间，AI Agent 编排框架快速扩张：

- **LangGraph**（~30K GitHub stars）
- **CrewAI**（多角色代理编排）
- **Google ADK**（事件驱动的代理框架）
- **OpenAI Agents SDK**（基于交接的编排）
- **Semantic Kernel**（微软企业级编排）
- **Strands**（AWS 模型驱动编排）
- **LlamaIndex Workflows**（事件驱动工作流）

这些框架加起来超过 29 万 GitHub stars。它们有一个共同特点：**在 LLM 上层放置一个外部编排器（orchestrator）**，每轮对话中向 LLM 注入指令、解析输出、决定下一步路由。

Dennis et al.（2026a）的研究表明，对于程序性任务，in-context prompting 实际上已经打败了这种编排架构——直接把完整流程塞进前沿模型的 system prompt，让模型自我编排，质量更高（4.53–5.00/5 分制）。

但 in-context prompting 有三个代价：

1. **上下文窗口消耗**：每轮都要把序列化后的流程图塞进 prompt，55 节点的保险理赔流程会导致约 7 倍的 token 体积膨胀
2. **每次对话都必须调用前沿模型**：成本高，无法本地部署
3. **专有流程暴露给第三方**：公司内部的工作流逻辑通过 API 调用流向了外部模型服务商

本文的核心主张是：**把这套流程编译进一个小模型的权重里**，训练出一个「地下代理」（Subterranean Agent），可以同时解决以上三个问题。

---

## 两种架构的对比

### 表面编排（Surface Orchestration）

```
用户 →编排器（Orchestrator）→ LLM
       ↑ 注入 prompt        ↓
       ← 解析输出 ←
```

- **运行时**：用户 → Orchestrator → LLM → Orchestrator → 用户（循环）
- 编排器每轮都要注入节点指令、解析 LLM 输出、决定下一个节点
- 路由决策由额外的 LLM 调用或规则引擎完成
- 流程逻辑以显式代码/配置存在，**不在模型权重里**

### 地下代理（Subterranean Agent）

```
用户 → LLM（权重已编译流程）
       ↑
   训练时：编排器生成训练数据
```

- **训练时**：用编排器（LangGraph + 前沿模型）生成大量合成对话，fine-tune 小模型
- **运行时**：用户直接和编译后的小模型对话，编排器彻底消失
- 流程逻辑被压缩进了模型的权重，推理时不再需要注入流程指令

## 实验设计：三个领域，三套对比基线

### 三个测试领域

| 领域 | 节点数 | 决策 hubs | 独特路径数 | 路径长度 |
|------|--------|-----------|-----------|----------|
| 旅行预订（Travel Booking） | 14 | 3 | 86 | 4–17 轮 |
| Zoom 技术支持（Zoom Support） | 14 | 3 | 60 | 4–17 轮 |
| 保险理赔（Insurance Claims） | 55 | 6 | 2,381 | 9–39 轮 |

**关键区分**：旅行和 Zoom 都是 14 节点的中等复杂度；保险理赔是 55 节点、近 4 倍规模的大流程，用来验证编译方案是否随复杂度增长而退化。

### 两套基线对比

- **LangGraph 编排器**：Claude Sonnet 4.5（~70 倍参数于 3B 模型）通过 LangGraph 框架编排，每轮决策都调用 LLM 做路由。这是**当前主流生产架构**的代表。
- **In-context 基线**：Claude Sonnet 4.5 + 完整的序列化流程图塞进 system prompt，每轮 self-orchestrate。代表**质量上界**——前沿模型 + 完整流程上下文。

### 评估方法

- **LLM-as-Judge**：Claude Sonnet 4.5 评分，不知道被评系统是哪个（盲评），5 个维度 1–5 分
- **独立验证**：用 GPT-4.1 重新评分以消除 Claude 自评偏好（83–99% 一致性结论）
- **n = 200**：每组条件每领域 200 个场景，覆盖完整路径分布、用户风格（具体到模糊）、满意度（热情到怀疑）

五个评分维度：

1. **任务完成率（Task Success）**：流程是否正确执行到合适的终止状态
2. **信息准确性（Info. Accuracy）**：提供的数字、日期、政策条款是否正确
3. **一致性（Consistency）**：前后表述是否矛盾
4. **优雅处理（Graceful Handling）**：遇到异常、拒绝、模糊时的表现
5. **自然度（Naturalness）**：对话是否像真人之间的自然交流

## Barrier 1：质量——小模型够用吗？

### Travel Booking 实验：同模型控制变量

这个实验的设计很干净。同样的 3B 基础模型（Qwen 2.5 3B Instruct），一组加编排器（3B Orchestrator），一组直接编译进权重（3B Subterranean Agent），其他条件完全相同。

| 指标 | 3B 地下代理 | 3B 编排器 | LangGraph Orch | In-Context |
|------|-----------|----------|--------------|-----------|
| 任务完成率 | **4.11** | 3.93 | 4.17 | 4.53 |
| 信息准确性 | **4.75** | 4.69 | 4.21 | 4.64 |
| 一致性 | **4.34** | 4.12 | 4.32 | 4.96 |
| 优雅处理 | **4.07** | 3.87 | **4.62** | 4.96 |
| 自然度 | **4.12** | 3.96 | **4.84** | 5.00 |

**结论**：编译后的 3B 地下代理在 5 项指标中 **4 项显著优于** 同样基础模型的编排器（p < 0.001）。编排器的节点级生成缺乏全局对话感知，容易出现重复提问；地下代理学会了端到端的流程感，信息准确性反而更高（4.75 vs. 4.69）。

对比 LangGraph Orchestrator（~70 倍大的前沿模型）：地下代理在信息准确性上领先（4.75 vs. 4.21），但在优雅处理和自然度上落后。说明 3B 模型学会了流程，但**缺少处理边缘情况的世界知识**。

### 质量差距分析

| 模型 | 信息准确性 | 优雅处理 | 自然度 |
|------|----------|---------|--------|
| 3B 地下代理 | **102%**（超预期） | ~82% | ~82% |
| 8B 地下代理 | ~87% | **~92%** | **~97%** |

3B 的主要短板是优雅处理和自然度——模型学会了「做什么步骤」，但缺乏处理「用户说不喜欢方案」「预算不够怎么办」这类情况的世界知识。**扩展到 8B 后这两个差距显著缩小**。

### Zoom Support（8B）：前沿编排器打成平手

| 指标 | 8B 地下代理 | LangGraph Orch | In-Context |
|------|------------|--------------|-----------|
| 任务完成率 | 4.50 | **4.62** | 4.92 |
| 信息准确性 | 4.26 | **4.75** | 4.92 |
| 一致性 | 4.42 | 4.55 | 5.00 |
| 优雅处理 | **4.62** | 4.52 | 5.00 |
| 自然度 | **4.87** | 4.64 | 5.00 |

8B 地下代理在**自然度和优雅处理**上**超越了** LangGraph Orchestrator（p < 0.001）。这意味着在产品技术支持这种需要自然对话引导的场景，一个编译进权重的 8B 模型比前沿编排器更流畅。

### Insurance Claims（8B，55 节点）：规模化压力测试

| 指标 | In-Context | LangGraph Orch | 8B 地下代理 |
|------|-----------|--------------|------------|
| 任务完成率 | 4.78 | 4.42 | **4.47** |
| 信息准确性 | 4.78 | 4.45 | **4.40** |
| 一致性 | 4.82 | 4.39 | **4.51** |
| 优雅处理 | 4.96 | 4.38 | **4.81** |
| 自然度 | 5.00 | 4.58 | **4.92** |

**关键发现**：编译方案成功扩展到了 55 节点、6 个决策 hub、2381 条路径的复杂工作流。8B 地下代理在 5 项指标中 **4 项超越** LangGraph Orchestrator，且达到了 In-Context 基线的 **92%–98%** 质量区间。

## Barrier 2：成本——便宜多少？

成本节省来自两个独立因素的叠加：

### 因子 1：每 Token 成本（~65×）

前沿模型 API 定价（Claude Sonnet 4.5）：
- Input：$3.00 / 1M tokens
- Output：$15.00 / 1M tokens

自托管 Qwen3-8B（A100 80GB，$2.50/hr via vLLM）：
- 有效费率：~$0.05 / 1M input tokens，~$0.23 / 1M output tokens
- **~65× 便宜**

### 因子 2：Token 体积（2–7×）

In-context 基线必须在**每轮对话**中都把完整流程图序列化进 system prompt，规模随流程复杂度增长：

- Travel（14 节点）：~2× token 体积
- Insurance（55 节点）：~7× token 体积

地下代理的提示词是固定大小的（最小 system prompt，如「你是旅行预订助手」），不随工作流规模增长。

### 叠加效应

| 领域 | In-Context | LangGraph Orch | 地下代理 | 性价比 |
|------|-----------|--------------|---------|--------|
| Travel（14 节点） | $0.133 | $0.077 | **$0.0010** | **128×** |
| Zoom（14 节点） | $0.103 | $0.054 | **$0.0003** | **296×** |
| Insurance（55 节点） | $0.327 | $0.174 | **$0.0007** | **462×** |

**最重要的结论**：流程越复杂，成本优势越大。55 节点的保险理赔场景比 14 节点的旅行预订贵了 3.6 倍（462× vs. 128×），因为 In-context 基线的 prompt 体积随流程规模膨胀，但编译模型的提示词始终是固定大小。

### 一次性编译成本

- 数据生成：~$40（Claude API 调用，并行化后几分钟完成）
- Fine-tuning：~$10–40（A100 单卡 3–4 小时）
- **总计：$50–80 一次性成本**

摊销到每轮对话：

- 500 次对话：每轮 < $0.16
- 10,000 次对话：每轮 < **$0.008**

**500 次对话即可收回编译成本**。

## Barrier 3：灵活性——流程变了怎么办？

这是开发者最担心的问题：「如果流程改了，小模型不是要重新训练？岂不是天大的工程成本？」

### 实际测量：30–50 分钟

论文测量了完整 recompile 周期（3 个阶段并行化）：

1. **数据生成**：Claude Sonnet 4.5 遍历新流程图，并行生成 ~1,600 个合成对话（API 调用天然可并行）
2. **训练**：Qwen 3B 单卡 RTX 5090，~3.5 小时；Qwen 3–8B 8×A100，~3–4 小时
3. **验证**：held-out eval loss 收敛检查

**端到端：30–50 分钟**（生产级硬件）

这不是「漫长的离线训练」，而是**一个 CI/CD 周期**——比很多团队的 sprint planning 会议还短。

### LoRA 为什么不工作

论文附录 Dennis et al.（2026b）系统研究了 LoRA：在 procedural 任务上，低秩微调方法（LoRA rank 16–128）无法接近全参数微调的质量。

**原因**：Procedural knowledge 不是风格对齐，而是改变模型的隐式状态跟踪行为——需要修改模型对「对话进行到哪一步」的深层表征，这不是低秩更新能学到的。

## 实验细节：训练是如何进行的

### 合成数据生成

每个领域的工作流被表示为流程图（有向图），包含：

- 节点（node）：每个节点是一个 prompt 模板，包含「在此节点代理说什么、用户可能如何回应」
- 边（edge）：节点间的转换，可带条件（如「用户接受 → 确认节点」）
- 起始节点（n₀）和终止节点（T：成功/放弃/升级）

生成流程：对每条路径采样 scenario 参数（目的地、预算、用户性格、险种类型等），用 Claude Sonnet 4.5 逐轮生成对话。

### 模型与训练配置

| 模型 | 精度 | 硬件 | 优化器 | 学习率 | Epochs | 最佳 checkpoint |
|------|------|------|--------|-------|-------|--------------|
| Qwen 2.5 3B Instruct | bf16 | RTX 5090 × 1 | AdamW 8-bit，batch=16 | 2×10⁻⁵ | 20 | epoch ~4 |
| Qwen3-8B | bf16 | 8×A100 80GB，ZeRO-3 | AdamW，batch=32 | 2×10⁻⁵ | 10 | epoch ~2 |
| Qwen3-8B（保险） | bf16 | 8×A100 80GB，ZeRO-3 | AdamW，batch=32 | 2×10⁻⁵ | 20 | epoch ~3 |

### 推理时的最小提示词

编译后模型的推理提示词：**不注入任何流程指令**。例如：

```
You are a helpful travel booking assistant.
```

没有 flowchart state，没有 routing logic，没有节点提示词——流程已经从「运行时注入的指令」变成了「模型权重里的隐式表征」。

## 失败率与对话风格

### 失败率对比

（失败 = 任务完成率评分 ≤ 3，满分 5）

| 领域 | 3B 地下代理 | 3B 编排器 | LangGraph Orch |
|------|-----------|----------|--------------|
| Travel | **5.5%** | 4.5% | 24.0% |
| Zoom | 11.0% | — | **9.0%** |
| Insurance | **9.0%** | — | 17.0% |

LangGraph Orchestrator 在 Travel 中的 24% 失败率主要来自**决策 hub 的路由错误**——在分支点，前沿模型可能选错下一条边。编译模型把这个决策编码进了权重，**消除了路由失败这个失败模式**。

### 对话粒度差异

地下代理从训练数据中学到了「**每次只问一个问题**」的节奏：64% 的代理轮次只包含一个明确问题。相比之下，LangGraph 编排器受节点模板限制，有时在一个轮次里塞进多个问题。

总词数相当（~1,200–1,400 词/对话），只是切成不同大小的轮次。更少的追问意味着**更清晰的审计追踪**，以及可能**降低用户的认知负担**。

## 核心结论：三个传统认知都被高估了

论文对三个传统认知一一做了量化反驳：

| Barrier | 传统认知 | 实证结果 |
|--------|---------|---------|
| **质量** | 小模型无法匹配前沿模型 | 8B 达到 87–98%，与 LangGraph Orch 相当 |
| **成本** | 本地部署不如调用 API | 128–462× 更便宜，且优势随复杂度增长 |
| **灵活性** | 重新训练是大型离线工程 | 30–50 分钟，是 CI/CD 周期，不是 paradigm shift |

核心设计哲学：**持久结构属于权重，瞬时状态属于上下文。**

## 适用边界与决策建议

### 什么时候选地下代理

- 流程相对稳定（不是每天都在改）
- 对话轮次多、context window 压力大
- 需要本地部署以满足数据隐私要求
- token 成本是主要瓶颈
- 流程是「程序性」的（步骤明确、分支清晰）

### 什么时候继续用编排

- 流程经常变化（需要每天 recompile）
- 需要在运行时插入实时数据源（外部 API 调用）
- 需要多模型协作（不同步骤用不同能力的模型）
- 任务以开放域推理为主（没有明确流程，模型需要自由探索）

### 如何开始

1. **先画流程图**：把当前用编排器表达的工作流画成有向图（节点 + 边 + 条件）
2. **生成合成数据**：用前沿模型 + 流程图生成 1,000–3,000 个对话样本
3. **全参数微调**：不要省这一步——LoRA 在 procedural 任务上不行
4. **30–50 分钟 recompile**：流程变更时走这个循环
5. **对比质量与成本**：跑 200 个场景，看质量是否达到 In-Context 的 87%+，成本是否真的省了 100 倍以上
