## 学习目标

学完本节后，你将能够：

- 理解 Claude 桌面应用里 Chat、Cowork、Code 三类工作方式各自负责什么。
- 知道什么任务适合继续留在普通对话，什么任务更适合交给 Cowork 或 Code。
- 了解桌面应用公开可确认的能力边界，避免把研究预览能力写成稳定能力。
- 建立“先选入口，再给权限，再逐步扩大自动化范围”的使用习惯。

---

## 先说结论

如果你只想先抓住重点，可以记住下面这句话：

> **Chat** 适合普通对话，**Cowork** 适合非编程的多步骤知识工作，**Code** 适合开发任务。

这三者不是“只是界面不同”，而是**自动化程度、可访问资源、风险边界**都不同。

你不应该把所有任务都丢给最强入口；更好的做法是：

1. 简单任务先从 Chat 开始。
2. 需要 Claude 持续处理文件、资料或周期性任务时，再考虑 Cowork。
3. 需要读仓库、改文件、跑命令、看 diff 时，再进入 Code。

---

## Claude 桌面应用到底解决什么问题

桌面应用的价值，不是“把网页搬到电脑上”，而是把 Claude 放进更接近真实工作的环境里。

和单纯的网页聊天相比，桌面应用更强调三件事：

| 维度 | Chat | Cowork | Code |
|------|------|--------|------|
| 核心目标 | 对话、问答、写作、解释 | 交付非编程的多步骤任务 | 完成开发相关任务 |
| 典型输入 | 一个问题、一段材料、一个草稿 | 一个结果目标 + 文件夹/资料/周期任务 | 一个代码库 + 一个开发目标 |
| Claude 的行动范围 | 以回答为主 | 可在授权范围内处理本地文件与长任务 | 可读项目、改文件、运行命令、做代码审查 |
| 风险级别 | 低 | 中到高 | 中到高 |
| 最适合谁 | 所有人 | 需要自动化知识工作的用户 | 开发者或技术团队 |

所以，Claude 桌面应用更像一个**多入口工作台**，而不是一个单一功能的聊天窗口。

---

## Chat：默认入口，先把问题说清楚

Chat 仍然是大多数人的起点。它最适合这些低摩擦任务：

- 快速问答
- 改写或润色一段文字
- 总结一份材料
- 解释一个概念
- 先做一版提纲或草稿

### Chat 的核心特点

1. **进入门槛最低**：不需要先理解项目、环境、权限模式这些概念。
2. **反馈最快**：你问一句，Claude 回一句，很适合摸清问题。
3. **适合先试再细化**：很多任务并不需要直接进入更重的自动化模式。

### Chat 不适合什么

如果你的任务具有下面特征，单靠 Chat 往往不够：

- 需要 Claude 长时间自己跑完整个任务
- 需要处理整个文件夹而不是单个片段
- 需要跨多轮长期积累项目记忆
- 需要真正进入代码库执行开发任务

换句话说，**Chat 更适合“先对齐需求”，不一定适合“代你执行全过程”**。

---

## Cowork：把 Claude 用在知识工作自动化上

根据 Anthropic 公开说明，**Cowork 是一项 research preview（研究预览）能力**，把 Claude Code 的 agentic 能力带到 Claude Desktop，但场景从编程扩展到了知识工作。

它更适合这类任务：

- 整理和重命名文件
- 从一堆资料中提炼研究结果
- 生成报告、文档、表格或演示材料
- 处理会持续一段时间的多步骤任务
- 设定按固定频率重复执行的任务

### Cowork 的几个关键点

#### 1. 它不是普通聊天的“加强版”

Cowork 的重点不是“回答更长”，而是你可以描述一个结果目标，让 Claude 在你授权的范围内完成多步骤工作。

#### 2. 它强调持续任务与工作区

官方资料明确提到，Cowork 支持：

- **projects**：把相关任务放进持续工作区
- **scheduled tasks**：让 Claude 按计划重复执行任务
- **global instructions / folder instructions**：给它长期偏好和目录级背景

这说明 Cowork 更像“面向知识工作的代理工作区”，而不是一次性会话。

#### 3. 它会接触真实文件，因此风险更高

官方帮助中心明确说明，Cowork 运行在你电脑上的虚拟机环境里，Claude 可以在你共享的文件范围内做真实更改。  
这也是为什么它比普通 Chat 更需要你认真管理权限和敏感数据。

### 什么场景适合优先选 Cowork

| 场景 | 为什么适合 Cowork |
|------|-------------------|
| 下载目录清理、资料归类 | 任务目标明确，且需要直接处理本地文件 |
| 周报 / 调研 / 汇总类任务 | 需要 Claude 长时间读材料、整合、输出成品 |
| 周期性固定工作 | 可以借助 scheduled tasks 自动重复执行 |
| 长期主题工作 | 可以放进 project，减少每次重复交代背景 |

### 什么场景不要急着上 Cowork

1. 只是问一个概念问题。
2. 只是润色一段短文。
3. 任务涉及高敏感个人或企业文件，但你还没想清权限策略。
4. 你自己都还没定义清楚目标，只是想“让 Claude 先做点什么”。

## Code：把 Claude Code 放进桌面图形界面里

Anthropic 的官方文档把 **Code tab** 描述为：在 Claude Desktop 中通过图形界面使用 Claude Code。

这意味着它的重点不是“普通聊天里顺便聊代码”，而是面向开发任务的完整工作流。

### Code 更适合什么任务

- 阅读和理解代码库
- 修改代码并查看 diff
- 运行命令、执行开发流程
- 结合 dev server 预览应用
- 跟踪 PR 状态、处理 CI 失败
- 在不同环境里运行开发任务

### 官方文档已明确公开的能力

根据 Claude Code Desktop 文档，目前公开写明的能力包括：

- **visual diff review**：用图形方式查看和评论改动
- **live app preview**：启动 dev server 并在预览面板里验证修改
- **computer use**：在桌面上操作应用和屏幕（默认关闭）
- **parallel sessions**：并行会话
- **scheduled tasks**：按计划运行任务
- **connectors**：连接 GitHub、Slack、Linear 等外部工具
- **local / SSH / cloud environments**：选择不同运行环境
- **permission modes**：控制 Claude 自动化权限范围

### 使用 Code 前，先理解 4 个配置项

官方文档把启动一个会话前的核心设置写得很清楚：

1. **Environment**：Claude 在哪里运行（本地、远程或 SSH）。
2. **Project folder**：Claude 工作的项目目录或仓库。
3. **Model**：会话使用的模型。
4. **Permission mode**：Claude 能自动执行到什么程度。

这 4 个概念比死记某个按钮更重要，因为它们决定了 Code 的真实工作边界。

## 三类工作方式怎么选

你可以把选择逻辑简化成一张表：

| 维度 | Chat | Cowork | Code |
|------|------|--------|------|
| 最核心的问题 | “你怎么看？” | “请把这件知识工作做完。” | “请在代码库里完成这个开发任务。” |
| 任务时长 | 短 | 中到长 | 中到长 |
| 是否依赖本地文件 | 可选 | 通常需要 | 通常需要 |
| 是否涉及代码库与命令 | 否 | 通常否 | 是 |
| 默认建议 | 先从这里开始 | 只有任务明显需要代理执行时再进入 | 只有任务明显是开发任务时再进入 |

### 一个简单判断法

1. **先问：这是不是开发任务？**  
   如果是，优先看 Code。
2. **如果不是开发任务，再问：Claude 是否需要自己连续做很多步，并处理你授权的本地资料？**  
   如果是，考虑 Cowork。
3. **如果以上都不是，Chat 往往已经够用。**

## 使用桌面应用时，最容易犯的 4 个错误

### 错误一：把所有任务都交给最强入口

这通常只会增加复杂度和风险。简单问题留在 Chat，往往更快。

### 错误二：把研究预览能力当成稳定承诺

Cowork 目前公开定位就是 research preview。对于这类能力，文档里最应该强调的是边界、权限和适用场景，而不是把每个细节都写成“稳定功能”。

### 错误三：还没想清权限就直接给太多访问范围

无论是 Cowork 还是 Code，真正的风险不在“AI 会不会说错”，而在“AI 会不会在你授权范围内做了真实动作”。所以权限设计必须比聊天更谨慎。

### 错误四：希望 Claude 自己定义目标

桌面代理能力更强，不代表它能替你完成目标澄清。目标不清时，先回到 Chat，把任务说清楚，再决定是否升级到 Cowork 或 Code。

---

## 一个更稳妥的入门顺序

如果你是第一次接触 Claude 桌面应用，建议按这个顺序上手：

1. **先用 Chat 做 3 个短任务**：总结、改写、解释。
2. **再试 Cowork 的一个低风险任务**：例如整理非敏感文件夹或生成一份简单汇总。
3. **如果你是开发者，再进入 Code**：从只读理解代码或小范围改动开始。
4. **最后再尝试更高自动化等级**：例如 scheduled tasks、computer use 或更宽的权限模式。

这个顺序的意义是：**先熟悉协作方式，再逐步扩大授权，而不是反过来。**

## FAQ

### Q1：桌面应用是不是一定比网页版强？

**答：** 不一定。对很多简单问题来说，网页版或普通 Chat 已经足够。桌面应用的优势，主要在于更贴近本地工作流，以及 Cowork / Code 这类更强的代理能力。

### Q2：Cowork 和 Code 的区别，最简单怎么理解？

**答：** 可以粗暴理解为：**Cowork 面向知识工作，Code 面向开发工作**。两者都比普通 Chat 更强调代理执行，但处理对象和工作环境不同。

### Q3：是不是一上来就该开最高权限？

**答：** 不建议。官方文档本身就把权限模式视为关键配置项。更稳妥的做法，是先从更保守的权限开始，确认任务目标和 Claude 的行为方式后，再逐步放宽。

### Q4：为什么这篇文章不再列一堆固定快捷键和界面按钮？

**答：** 因为这类细节更容易随产品更新而变化。对于入门文档，更重要的是说明公开可验证的能力边界、入口差异和选择逻辑，而不是堆一批容易过期的界面细节。

## 行动建议：今天就能做的 3 个动作

1. **先用 Chat 做一个真实小任务**，感受 Claude 的基本协作方式。
2. **挑一个低风险、非敏感的文件类任务试 Cowork**，体验“描述结果目标”而不是“逐句聊天”。
3. **如果你是开发者，用 Code 做一次小改动或代码解释**，重点观察 environment、project folder 和 permission mode 这三个概念。
