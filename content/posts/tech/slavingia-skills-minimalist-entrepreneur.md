## 导读

如果你把这篇文章读完只带走一句话，那应该是：这套 Skills 的重点不是帮你更快写出一个产品，而是帮你更早验证“这个产品到底该不该存在”。

## 1. 学习目标

读完这篇文章后，你应该能够：

- 准确理解 Minimalist Entrepreneur Skills 的定位，而不是把它误解成“通用创业话术库”。
- 知道 10 个 Skill 各自的边界、前置条件和最适合的使用时机。
- 理解这套方法论为什么强调“先手工交付，再逐步产品化”。
- 用一条清晰的流程，把找社区、验证问题、做 MVP、拿前 100 个客户串起来。
- 识别这套方法最容易被误用的地方，并知道如何规避。
- 根据自己的创业阶段，快速选出下一步最值得调用的 Skill。

## 2. 先说结论：这套 Skills 适合谁，不适合谁

Minimalist Entrepreneur Skills 是 Sahil Lavingia 的公开仓库 slavingia/skills 中的一组 Claude Code Skills。它们基于《The Minimalist Entrepreneur》这本书的方法论，目标不是替你“生成创业点子”，而是把早期创业里最容易自我欺骗的环节，拆成一系列更难逃避现实的问题。

它最适合三类人：

- 已经想做点东西，但不知道应该先找人还是先写代码的人。
- 有产品冲动、工程能力强，但用户验证和销售动作明显偏弱的人。
- 想把 Claude Code 从“写代码助手”升级成“产品验证与经营决策助手”的人。

它不太适合以下场景：

- 你要的是融资建议、法律建议、税务建议。
- 你已经是成熟公司，需要的是组织流程、复杂渠道和规模化经营体系。
- 你还没有任何社区连接，却期待它直接给你一个高确定性的商业机会。

一句话概括：这不是“帮你做大公司”的技能包，而是“帮你别在第一步就走歪”的技能包。

## 3. 它的核心方法论，为什么成立

这套 Skills 的底层逻辑可以压缩成 4 句话：

1. 先从社区开始，不要先从点子开始。
2. 先验证有人痛，再验证有人付费。
3. 先手工交付价值，再把稳定流程产品化。
4. 先追求盈利和生存，再谈增长和规模。

这套方法成立，不是因为它更“优雅”，而是因为它直接对抗了早期创业最常见的 4 个错觉。

### 3.1 错觉一：我先把产品做出来，再去找用户

这正是多数早期项目最致命的路径依赖。你一旦先投入大量构建成本，就会天然为既有方案辩护，而不是重新判断“这个问题值不值得解”。

Minimalist Entrepreneur 的反向做法是：先找到一个你真正理解的社区，再看这个社区里有没有持续出现、愿意付费解决的问题。

### 3.2 错觉二：只要用户说“这个想法不错”，就算验证过了

不是。愿意夸你的人，不等于愿意买你的人。

这套 Skills 反复强调一个原则：验证不是通过“点赞”，而是通过“交易”完成。只要还没有人愿意付钱，你得到的大多数都只是情绪支持，而不是商业信号。

### 3.3 错觉三：MVP 就是做个简陋版产品

很多人把 MVP 理解成“功能少一点的软件”。但这套方法论把 MVP 放在更靠前的位置看待：真正的最小可行，不是“最少功能”，而是“最早开始交付价值的方式”。

所以它才会强调 manual first，也就是先手工交付，再流程化，再自动化。

### 3.4 错觉四：增长越快越好

这套 Skills 对增长持非常克制的态度。它的核心不是追逐速度，而是保证你在增长时没有把自己拖进高成本、低控制权、不可逆的局面。

换句话说，它优先优化的是“默认活着”，而不是“看上去很大”。

## 4. 这 10 个 Skills 到底是什么

仓库当前提供 10 个 Skill，覆盖从找社区到复盘决策的完整链路：

| Skill | 命令 | 最适合解决的问题 |
| ------ | ------ | ------------------ |
| Find Community | /find-community | 我该服务谁，而不是我想做什么？ |
| Validate Idea | /validate-idea | 这个问题真的值得做吗？ |
| MVP | /mvp | 我到底应该做多小，才能在本周交付？ |
| Processize | /processize | 在写代码前，怎样先手工交付价值？ |
| First Customers | /first-customers | 怎样拿到前 100 个客户？ |
| Pricing | /pricing | 我应该怎么收费，才既有信号又可持续？ |
| Marketing Plan | /marketing-plan | 当我已有一定验证后，如何用内容做增长？ |
| Grow Sustainably | /grow-sustainably | 这笔钱该不该花、这人该不该招、这步该不该扩？ |
| Company Values | /company-values | 我到底想建一家什么样的公司？ |
| Minimalist Review | /minimalist-review | 任何复杂决策，怎样用极简创业原则复核？ |

如果你只记一个判断标准，那就是：前 6 个 Skill 更偏“验证与成交”，后 4 个 Skill 更偏“增长、经营与决策”。

## 5. 每个 Skill 的真实作用与使用边界

### 5.1 Find Community：先找“人群”，不是先找“创意”

这个 Skill 的核心原则是 community first。它会逼你回答：你已经属于哪些社区、你在哪些地方长期出现、你反复听到哪些抱怨、你愿不愿意服务这群人很多年。

为什么它重要？因为脱离社区想出来的机会，通常只有概念，没有分发，也没有真实痛感。你缺的不是点子，而是上下文。

什么时候该用：

- 你点子很多，但没有一个能明确说出“谁会先买”。
- 你经常从技术能力出发思考，而不是从人群出发思考。
- 你对目标用户的聚集地、语言和习惯并不熟。

什么时候不该用：

- 你已经有明确的社区连接和一批具体潜在用户。

最容易犯的错：

- 把“大市场”误认为“好市场”。
- 把“我能接触到”误认为“我真正属于”。
- 直接跳到“我能卖什么”，而不是“他们反复卡在哪里”。

### 5.2 Validate Idea：验证的对象不是功能，而是痛点与付费意愿

这个 Skill 不会鼓励你先做原型，而是先定义问题：谁在痛、现在怎么绕过去、痛到什么程度、有没有现成替代方案、有没有人愿意现在就掏钱。

它的重要性在于，它把验证标准从“用户觉得不错”升级成“用户会不会给你交易信号”。

优先关注的不是“市场有多大”，而是：

- 你能否说出 10 个具体的人。
- 他们今天是否已经在用笨办法解决问题。
- 你能否立即手工提供一个版本。
- 是否已经有人愿意为这个结果付费。

红旗信号通常包括：

- 没有人正在用替代方案。
- 你说不出 10 个具体潜在客户。
- 你必须先教育用户“你其实有这个问题”。
- 唯一验证依据是“朋友说这个想法很酷”。

### 5.3 MVP：不是“功能压缩”，而是“本周能交付什么”

这个 Skill 解决的是范围失控问题。它不是帮你列一个产品 PRD，而是逼你回答 4 个更现实的问题：

- 能不能在一个周末交付？
- 能不能让客户的生活哪怕只好一点点？
- 能不能从第一天开始收费？
- 能不能很快拿到反馈？

它和 Processize 的区别，经常被写反。

简单说：

- MVP 关注“最小交付结果是什么”。
- Processize 关注“这个结果在手工状态下如何稳定交付”。

如果你已经知道要解决什么，但迟迟收不住功能范围，就该先用 /mvp。

### 5.4 Processize：这是整套方法里最容易被低估的 Skill

这是本文最值得强调的一点。当前仓库里，/processize 是一个非常关键的桥梁：它要求你在写任何代码之前，先把手工交付路径走通。

它会让你写清楚：

- 用户今天给你什么输入。
- 你手工做哪些步骤。
- 你最终交付什么输出。
- 每一步要花多久。
- 哪一步最适合未来自动化。

为什么这一步价值极高？因为只有当你真的人工交付过，才知道流程里哪些部分是稳定需求，哪些只是你脑补出来的系统复杂度。

一句更直白的话：很多人不是太早产品化，而是太早系统化了。

### 5.5 First Customers：前 100 个客户不是“等来”的，是一个个卖出来的

这个 Skill 的核心观点非常明确：别迷信 launch，先专注 sell。

它强调一个由近及远的销售顺序：

- 先朋友和家人。
- 再你所属社区里认识你的人。
- 最后才是冷启动外联。

这背后的原因很现实。早期客户获取不是品牌问题，而是信任问题；不是流量问题，而是转化前的对话问题。

如果你连最接近你的人都无法说服，问题大概率不在流量，而在价值表达、目标用户或产品本身。

### 5.6 Pricing：收费不是商业化后期动作，而是验证动作

这个 Skill 强调一件很多人不愿接受的事：免费几乎不给你有效信号。愿意免费试的人，和愿意付费解决问题的人，往往不是同一类人。

它主要帮你回答：

- 你的成本结构是什么。
- 用户感知价值大概在哪。
- 怎样定一个足以形成信号、又不至于把用户吓跑的初始价格。
- 你需要多少客户，才足以支撑自己。

如果你当前还在说“等产品成熟了再收费”，那通常意味着你还没有真正验证价值。

### 5.7 Marketing Plan：它不是“做宣传”，而是把销售经验扩展成内容系统

这个 Skill 有一个很重要的前提：通常在你已经接近产品市场契合，或至少已有一批真实客户之后再用更合适。

它的核心不是广告，而是内容。不是“曝光越多越好”，而是通过内容让更多正确的人理解你、记住你、信任你。

它把内容分成 3 个层次：

- Educate：教别人你学到的东西。
- Inspire：分享你的过程、挣扎和变化。
- Entertain：让内容更容易被传播。

这部分最常见的误用，是在没有销售经验、没有客户语言、没有核心问题画像时，就急着“做内容矩阵”。那样内容通常会空心化。

### 5.8 Grow Sustainably：这是经营层面的刹车系统

如果说前面的 Skill 负责让你起步，那么 /grow-sustainably 负责防止你在稍微有点势头后开始自毁。

它围绕几个高风险决策展开：

- 要不要花这笔钱。
- 要不要招这个人。
- 要不要搬办公室。
- 要不要融资。
- 要不要为了看起来更快而牺牲盈利能力。

它真正提供的是一种经营视角：利润不是保守，而是控制权；低成本不是寒酸，而是战略空间。

### 5.9 Company Values：价值观不是口号，而是组织决策的约束条件

这个 Skill 不是给你生成 4 个好听词汇，而是逼你回答一些更难的问题：

- 你相信什么，是大多数人未必同意的？
- 什么行为即使短期有业绩，你也不能接受？
- 什么行为即使暂时不赚钱，你依然愿意鼓励？

也就是说，价值观在这里不是品牌装饰，而是组织边界。

### 5.10 Minimalist Review：用统一原则复盘所有复杂决策

这是一个通用复核器。只要你陷入“这件事好像有很多理由要做”的状态，就适合拿它来过一遍。

它关注的问题包括：

- 这件事是否真的服务客户或社区。
- 有没有更简单、更便宜、更可逆的版本。
- 现在做是不是因为客户需要，还是因为自我感觉更像一家“大公司”。
- 这一步会让你更接近盈利，还是更远离盈利。

如果你只能保留一个 Skill 用在长期经营上，那很可能就是它。

## 6. 正确使用顺序：不要把 10 个 Skill 当成 10 个孤立工具

这套仓库真正强的地方，不在于单个 Skill，而在于它们能串成一条路径。

### 6.1 推荐路径

| 阶段 | 你要解决的核心问题 | 推荐命令 |
| ------ | ------------------ | ---------- |
| 找方向 | 我到底该服务谁？ | /find-community |
| 做判断 | 这个问题值不值得做？ | /validate-idea |
| 收范围 | 本周到底做什么最小交付？ | /mvp |
| 先交付 | 不写代码，怎么先把价值交出去？ | /processize |
| 拿成交 | 怎样一个个拿到付费客户？ | /first-customers |
| 设价格 | 收费应该怎么起步？ | /pricing |
| 做扩散 | 已有基础验证后，如何做内容增长？ | /marketing-plan |
| 控节奏 | 什么时候花钱、招人、扩张？ | /grow-sustainably |
| 定文化 | 要建什么样的公司？ | /company-values |
| 做复盘 | 复杂决策怎么回到原则？ | /minimalist-review |

### 6.2 最容易出问题的错误顺序

错误顺序通常长这样：

Marketing Plan → MVP → Pricing → Find Community

为什么错？因为你在没确认服务对象、没确认问题强度、没确认付费信号之前，就已经开始想增长、做产品和谈价格。这会把后面的每一步都建立在不稳定前提上。

## 7. 安装与调用：哪些写法是对的，哪些写法是错的

根据上游仓库 README，最直接的安装方式是在 Claude Code 中执行：

```text
/plugin marketplace add slavingia/skills
/plugin install minimalist-entrepreneur
```

如果你想从本地克隆安装，仓库给出的思路是先克隆到本地插件目录，再通过 marketplace add 指向本地路径，而不是简单复制某个 skills 子目录。

示例：

```bash
git clone https://github.com/slavingia/skills.git ~/.claude/plugins/skills
```

然后在 Claude Code 中执行：

```text
/plugin marketplace add ~/.claude/plugins/skills
/plugin install minimalist-entrepreneur
```

这里有两个常见误区：

- 误区一：把它理解成“复制几个 Markdown 文件就能直接用”。
- 误区二：把本地安装写成随意拷贝某个嵌套目录。

更稳妥的理解是：这是一个通过 Claude Code 插件与 Skill 机制分发的技能集合，直接按仓库 README 的安装方式走，最不容易出错。

## 8. 一个更实战的理解方式：把它当成创业流程编排器

如果你是做 AI 工具、自动化产品或独立开发项目，这套 Skills 最有价值的地方，不是给你结论，而是让你每一步都更难自欺。

> 延伸阅读：如果你想看一条完整的落地路径，可以继续读 [如何用 10 个 Minimalist Entrepreneur Skills 跑完一个真实约束创业案例]({{< relref "slavingia-skills-real-startup-case.md" >}})。

比如你想做一个“帮内容创作者自动拆解长视频并生成短视频脚本”的产品，比较合理的使用顺序会是：

1. 先用 /find-community，明确你服务的是哪一类创作者，而不是泛泛地说“内容行业”。
2. 再用 /validate-idea，确认他们今天是否已经在手工切片、是否觉得这个过程真的痛、是否有预算。
3. 再用 /mvp，先把“到底交付什么结果”收缩到一个本周能推出的最小版本。
4. 接着用 /processize，先手工接 3 个创作者案例，自己把交付流程跑通。
5. 然后用 /pricing，给出一个能收钱的初始交付方式。
6. 再用 /first-customers，把早期成交动作系统化。
7. 最后在有稳定客户后，再考虑 /marketing-plan。

这条路径的关键不是“快”，而是每一步都让下一步更便宜、更清楚、更可逆。

## 9. 这篇文章原本最需要修正的几个认知点

为了避免你读完仍然带着错误理解离开，这里把几个最关键的纠偏点单独列出来。

### 9.1 Processize 不是附属步骤，而是整套方法的核心桥梁

很多介绍会把 /processize 写成一个可有可无的补充步骤，但从上游仓库当前内容看，它实际上承担着“把想法变成可手工交付流程”的关键作用。这个 Skill 的加入，让整套路径从“概念正确”变成“动作可执行”。

### 9.2 Marketing Plan 有明显前置条件

它不是“刚有个想法就该用”的 Skill。仓库写得很清楚，它更适合在已有产品市场契合迹象，或者至少接近约 100 个客户、已经理解销售过程后使用。

### 9.3 Pricing 不该被当成后置工作

如果你总把定价放到最后，本质上是在逃避最重要的商业验证问题：有没有人愿意为结果付费。

### 9.4 这套 Skills 不是替代判断，而是强化判断

它能帮你提出高质量问题、暴露你回避的问题、生成结构化输出，但它不能替你承担用户访谈、销售对话、定价试错和经营责任。

## 10. 最佳实践：怎样用，产出才会明显更好

### 10.1 把它当“提问器”，不是“答案机”

如果你对问题输入得很空，它输出也很难有高价值。越具体的上下文，越能得到像顾问而不是像模板的结果。

### 10.2 每次只解决一个阶段问题

不要一次让 Claude Code 同时帮你找社区、做定价、写营销计划、起公司价值观。那样看似全面，实际会把每一步都做浅。

### 10.3 保存每次输出，形成自己的经营文档

这类 Skill 的真正复利，不在单次对话，而在你把每轮输出沉淀下来，反复比较：哪些判断变了、哪些假设被证伪、哪些信号开始稳定。

### 10.4 对任何“听起来很顺”的答案，继续追问证据

最危险的不是差答案，而是顺滑但未经验证的答案。只要涉及市场、定价、用户意愿、增长速度，就一定要把“证据是什么”问到底。

## 11. 自测：你是否真的理解了这套 Skills

如果你想确认自己不是“看懂了，但不会用”，可以先回答下面 6 个问题：

1. 你能说出自己已经属于的 3 个社区吗？
2. 你能列出 10 个具体潜在用户，而不是抽象用户画像吗？
3. 你现在准备做的产品，能否用手工方式先交付给 3 个人？
4. 你是否已经有一个可以直接报价的初始方案？
5. 你现在最该做的是销售，还是增长？
6. 如果这周不能写代码，你还能不能开始验证这件事？

如果你对其中 3 个以上问题回答不清楚，那说明你更需要这套 Skills，不是更不需要。

## 12. 常见问题

### Q1：这套 Skills 和直接跟 Claude 普通聊天，有本质区别吗？

有。区别不在“模型更强”，而在它把一类问题固定成了更稳定的结构和追问路径。普通聊天容易跑偏，Skill 更像预先设计好的思考框架。

### Q2：一定要按顺序把 10 个 Skill 全用一遍吗？

不需要。但如果你还在创业早期，至少应该优先保证前半段顺序基本正确，也就是先社区、再验证、再最小交付、再成交。

### Q3：MVP 和 Processize 会不会重复？

会有交叉，但关注点不同。/mvp 更关注“做多小”，/processize 更关注“先怎么手工做”。前者收范围，后者走流程。

### Q4：如果我是技术很强、销售很弱的人，最该先用哪个？

大多数情况下，先从 /find-community 或 /validate-idea 开始，而不是从 /mvp 开始。因为你的主要风险不是做不出来，而是做对谁、为什么值得做。

### Q5：这套 Skills 支持中文吗？

Skill 内容本身主要是英文，但你完全可以用中文与 Claude Code 互动。关键不是语言，而是你给出的事实是否具体、是否真实。

## 13. 一页行动清单

如果你今天只打算做一件推进创业的事，按下面顺序执行：

1. 写下你已经真正属于的 3 个社区。
2. 在每个社区里列出 3 个反复出现的问题。
3. 选出一个你能立刻手工帮别人解决的问题。
4. 给这个手工结果报一个具体价格。
5. 本周联系 3 个具体的人，尝试交付并收集反馈。
6. 只有当手工流程开始稳定时，才进入工具化或产品化。

## 14. 总结

slavingia/skills 的价值，不在于它把创业讲得更热血，而在于它把创业拆得更现实。

它最强的地方有三个：

- 它强迫你从社区和问题开始，而不是从功能和幻觉开始。
- 它强迫你把“验证”落实为可收费、可交付、可复用的动作。
- 它给了你一套从想法、销售、定价到经营决策的统一原则。

如果你会写代码，但总是在“做什么、先做什么、怎么证明值得做”这几个问题上消耗过多，这套 Skills 值得认真用起来。

真正的重点不是把 10 个命令都试一遍，而是让自己形成一个新习惯：任何产品冲动出现时，先问社区、再问交易、再问手工交付、最后才问系统实现。

这才是 Minimalist Entrepreneur Skills 最值得带走的东西。
