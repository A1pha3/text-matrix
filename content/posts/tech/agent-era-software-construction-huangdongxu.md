# Agent时代软件构建革命：黄东旭百亿Token实践后的深度思考

> **目标读者**：软件工程师、技术决策者、AI研究者、对Agent时代软件发展趋势感兴趣的所有人
> **前置知识**：软件工程基础、了解大语言模型和Agent概念
> **难度定位**：⭐⭐⭐⭐ 专家设计
> **预计阅读时间**：45-60 分钟

---

## §1 学习目标

完成本篇文章后，你将能够：

1. **理解代码本质的转变**：为何"代码即执行载体，思考媒介已变"
2. **掌握Agent软件生产的三维框架**：Goal / Context / Constraints的分工与价值
3. **看清软件形态的演进方向**：从静态UI到动态"意图驱动"
4. **理解人机协同的正确姿势**：不是当甩手掌柜，而是做"HR+架构师"
5. **把握产业格局变化**：为何中间层SaaS会被严重压缩
6. **预判下一代抽象**：Skill为何不够、下一代是什么

## §2 背景：一个CTO与他的Agent Team

### 2.1 作者背景

黄东旭，TiDB联合创始人兼CTO，工程师出身，至今仍每日写代码。

> "我不再开会了，所有的时间都在跟我的Agent开会。这几周算是'出关'了。"

过去几个月，他完全用AI Coding Agent开发了一个**几百万行代码的基础软件项目**（类似编译器的数据库项目），**100%无人工，目前已在生产环境中线并有了客户**。

### 2.2 百亿Token的代价

黄东旭透露，他燃烧了**上百亿Token**来做这些事情。但他强调：

> "大家不应该把这个作为目标去看。真正的目标应该是去完成了一些有意义的事情，然后再回头审视，在这个旅程中你得到了什么。"

这是一个重要的价值观提醒：**Token消耗量不是成就指标，交付成果才是**。

---

## §3 核心观点一：Coding已死，软件迎来寒武纪大爆发

### 3.1 判断的依据

黄东旭的判断基于一个关键时间点：**2024年11月**。

> "在那个时间点之前，你可以认为所有的Coding Agent对程序员做复杂工程只能起到辅助作用。但在那个时间点之后，它跨过了某一个极限的阈值——系统可以完全自主地去完成长程、复杂的软件任务，且交付的结果已经可以用了。"

这个判断的核心逻辑：

```
任意复杂的软件工程 → 可以拆解为局部不那么复杂的模块
局部模块的完成能力已经Ready → 模型+Agent可以完成任意复杂度的软件
```

### 3.2 代码本质的转变

在传统软件开发中，代码有双重角色：

| 角色 | 说明 |
|------|------|
| **执行载体** | 代码在硬件上被执行 |
| **思考载体** | 程序员用代码在头脑中构建和表达思想 |

**但Coding Agent改变了一切**：

> "几个月前，我们作为程序员还在用代码去思考；但今天，我已经完全不再看代码了，AI在生成什么东西我完全不看。那代码变成了什么？代码变成了跟硬件一样的执行载体。"

这意味着：

```
人类思考：Goal + Context + Constraints（用自然语言）
     ↓
AI执行：代码（纯粹的执行载体）
```

**人类最应该关注的是Goal——我们到底要什么**。这是人类最擅长、AI最难替代的部分。

### 3.3 软件行业的未来

很多人担心AI会让软件行业萎缩。黄东旭的判断恰恰相反：

> "软件这件事情其实反而会迎来一种类似'寒武纪大爆发'的状态。"

原因：上一次这种级别的载体变革（编程语言的发明）诞生了整个软件行业；这一次，它会产生什么？这还是一个**Open Question**。

## §4 Agent软件生产的三维框架

### 4.1 框架总览

黄东旭提出了一个简洁但强大的框架：**Goal / Context / Constraints**。

```
┌─────────────────────────────────────────────────────────────┐
│                  Agent软件生产的三个维度                      │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│   Goal（目标）     │   Context（上下文）   │  Constraints（约束）│
│   ─────────────   │   ────────────────   │  ────────────────  │
│   人类最应关注     │   给Agent充足的信息   │  控制边界和风险     │
│                   │                      │                    │
│   • 人机交互设计   │   • Memory           │   • Sandbox        │
│   • Skill生态     │   • RAG              │   • Agent测试环境   │
│   • 意图理解      │   • Context Window   │   • CICD           │
│                   │   • Harness Eng.     │   • Lint           │
│                   │                      │                    │
│   [自然语言/对话]  │   [信息供给]          │   [控制与安全]      │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 Goal维度

**人类最应该投入精力的地方**。

但这里有一个关键的洞察：

> "人类其实有一个普遍的逻辑弱点——经常不知道自己到底要什么。这对于AI来说是特别麻烦的事情。"

因此，Goal维度的核心工作不是写代码，而是：
- **精确描述需求**：把模糊的想法变成清晰的目标
- **定义验收标准**：如何判断"做完了"
- **分解大目标**：把一个宏伟愿景拆成可执行的小目标

目前火热的Skill和各种人机交互产品，其实都在解决Goal维度的问题。

### 4.3 Context维度

**如何让Agent拥有足够的信息来完成任务**。

黄东旭提到的技术包括：
- **Memory（记忆）**：让Agent记住历史上下文
- **RAG（检索增强生成）**：从知识库中检索相关信息
- **Harness Engineering**：把多个Agent放在一个环境里协同工作
- **Context Window优化**：在有限窗口内塞入最关键的信息

Context维度是当前最卷的方向，各种创业公司都在做。

### 4.4 Constraints维度

**控制边界和风险**——这是当前Agent Infra最火的赛道。

黄东旭观察到：
- **Sandbox**：隔离环境，保护系统安全
- **Agent测试环境**：验证Agent行为是否符合预期
- **CICD**：自动化测试和部署
- **Lint**：代码质量和风格检查

本质上，Constraints就是在给AI套上"缰绳"，让它在可控范围内发挥。

## §5 人机协同的正确姿势

### 5.1 从"甩手掌柜"到"HR+架构师"

黄东旭分享了他自己的思维转变过程：

**第一阶段（激进）**：完全黑盒运行，人不干预，让Agent自己讨论、开发、消耗Token。他的信念是：

> "The more tokens, the more intelligence."

**结果**：不太Work。

**第二阶段（调整后）**：人还是要在流程里"点拨"一下。现在他跟Agent Team的工作模式更像是**HR**：

> "我每天在群里让大家签到，总结昨天干了什么，谁表现得最好让大家学习，谁表现得不好让他反思。"

### 5.2 Spec的重要性

一个深刻的教训：

> "无人监督时模型容易跑飞，此时一份写得非常好的Spec（需求规格说明书），远比单纯把多个Agent放在一个群里聊天的Harness Engineering要重要得多。"

这是朴素的软件工程经验在Agent时代的价值：**好的Spec比多Agent协作更重要**。

### 5.3 大力出奇迹

黄东旭的实践是：对于任何需求，**强制要求数十个Agent不停地、一遍又一遍地重写、删改并互相讨论**。

> "可以说99%的时间它们产出的都是废话，但我要的就是那1%灵光一闪的Idea。这种时刻出现时，你会发现整个代码质量会发生直接的跃迁。"

**核心洞察**：想做个好软件还要省Token，是不行的。**必须要大力出奇迹**。

### 5.4 模型智商是决定性因素

> "在成千上万步的长程任务里，智商差1%，跑到后面肯定就飞了。"

用智商低的模型省钱，在长程任务里一定会后悔。

## §6 软件形态的演进

### 6.1 从静态到动态

在LLM和Agent出来之前，软件形态是**静态的**：
- UI是固定的
- 功能是固定的
- 软件的智商上限 = 程序员和产品经理的智商上限

**但今天，软件已经没有固定形态了。**

> "今天其实已经没有什么UI和固定形态了，'意图'才是用户未来操作软件的真正接口。"

这意味着：**用户说"我想做什么"比"怎么去做"更重要**。

### 6.2 OpenClaw定义的"60分及格线"

黄东旭特别提到了OpenClaw（小青龙/小龙虾）：

> "OpenClaw最大的意义，在于它定义了Agent时代软件的'60分及格线'。如果未来的新软件，体验和灵活性不如OpenClaw，那就是旧时代的东西。"

这是一个非常重要的标准：**未来的软件，体验必须超过OpenClaw，否则就是落后的**。

### 6.3 Unix哲学的复兴

黄东旭是Unix哲学的忠实粉丝。他指出：

> "比起固定的图形Dashboard（如果想看新指标只能重新开发界面），Unix哲学里改需求就是把grep error改成grep warning这么简单。你不需要从头开发新软件。大模型只要学会了Unix哲学和常用Unix Tool，就能满足各种需求。"

这解释了为什么CLI和Unix哲学在Agent时代重新火了起来。

## §7 产业格局的重组

### 7.1 产业链被压缩

黄东旭画了一根轴，描述了产业链的压缩：

```
[极度贴近人] ←————————————→ [极度贴近机器]
    │                                    │
    │   情绪价值 + 意图驱动              │   编译器/数据库/OS
    │   交付结果决定留存                  │   无账号体系/纯基础设施
    │                                    │
    │   [中间SaaS和行业软件被严重压缩]    │
```

**预测**：在Agent普及后，产业轴会被严重压缩，创新和资本会往两头走：
- **靠近人的一端**：看"情绪价值给满"——让用户感觉"I'm in control"
- **靠近机器的一端**：看"交付结果是否扎实"——基础软件不再是给人用的

### 7.2 账号体系的消亡

一个具体例子：TiDB Cloud的新产品线叫**Zero**，特点是**没有账号体系**。

> "有账号体系的意思就是说人要登录，要去鉴权，但是Agent的数量和需求可能是人的100倍、1000倍、10000倍，这时候你再用人的账号体系去限制Agent其实是不明智的。"

这背后是一种思维变化：**把自己变成纯粹的招之即来、挥之即去的基础设施**。

### 7.3 "Everything is Skill"

> "包括现在我的数据库的onboarding，其实就已经完全不是说让人安装一个什么东西了，把一段Skill贴进对话框，它自己就跑起来了。没有注册，没有Configuration，Everything is Skill。"

这是一个方向：**未来的软件交付，可能就是一段Skill**。

## §8 Agent体工学：下一波热潮

### 8.1 新行业诞生

黄东旭预判：

> "马上也会有一个专门针对Agent使用软件的行业出现——怎么去给Agent设计更好用的软件。"

这个新行业，他称之为**"Agent体工学"**——类似于传统的"人体工学"（如何让人更舒服地使用工具），但现在是"如何让Agent更舒服地使用软件"。

### 8.2 三大原则

| 原则 | 说明 |
|------|------|
| **最小化使用的摩擦** | Agent不需要在多个系统之间切换。Unified Platform比分散的工具链更好 |
| **最大化的信息密度** | Markdown比HTML信息密度高，Agent更喜欢结构化的表达 |
| **不相信自己的流程** | 人类自己想出来的流程规范，放在Agent面前可能都是愚蠢的 |

## §9 Skill的局限与下一代抽象

### 9.1 Skill太简单

黄东旭认为Skill只是过渡方案：

> "Skill核心问题在于太简单、太粗糙了。今天我们交换Skill的方式，就跟当年交换奇奇怪怪的recipe的方式是一样的。但是未来像AWS或者Photoshop等庞杂的软件，是不可能通过'一张纸'去交付的。"

### 9.2 下一代抽象的特征

未来在Skill之后一定会有更好的抽象。这个抽象需要满足：

1. **能表达更复杂的协同**：多步骤、多角色、多系统的协调
2. **通过自然语言人机协同生成**：人类可以用自然语言指导生成
3. **有一定的结构且能被Debug**：不是黑箱，有可解释性
4. **核心目标：蒸馏人类专家的知识**：把专家的Know-How变成可复用的资产

### 9.3 软件开发门槛被击穿

> "比如我今天是个厨师，以前受限于开发门槛没法做做菜的APP；但今天只要跟Agent聊上三天三夜，把我的Know-How蒸馏出来变成一个'Cooking Stack'，就可以变现了。"

这意味着：**未来的软件可能会变成"Stack"而不是"App"**——一组经过蒸馏的专家知识，用自然语言可以调用。

## §10 战略格局：为何Anthropic赢了

### 10.1 不是科学规律，是战略路径

黄东旭提出了一个深刻的观点：

> "为什么看起来是Anthropic赢了？这不是因为发现了新的科学规律，而是因为这就是目前的战略路径。如果Google赢了，世界可能是纯多模态的；但今天是Anthropic赢了，所有的资金和算力都在疯狂加强模型的编程能力。"

### 10.2 通用Agent的基石

> "世界变成了一个'以编程为通用手段来改变一切'的形态，未来的所有通用Agent都将构建在编程能力这个基石之上。"

这是一个结构性的判断：**如果你相信未来的Agent会改变一切，那么你就必须相信编程能力是底座中的底座**。

## §11 朴素工程经验的价值

### 11.1 Bootstrap阶段的人类遗产

黄东旭提到：

> "我们这个行业真的太爱发明一些稀奇古怪的词了。Harness engineering你翻译过来看，它就是Still Good Engineering，它没有什么特别的。"

Agent时代，我们能给Agent最好的东西，其实就是**上一个时代积累下来的朴素软件工程、复杂性管理、需求分析、项目管理这些东西**。

### 11.2 给Agent的遗产

| 遗产 | 对Agent的价值 |
|------|---------------|
| 软件工程 | 构建可维护、可扩展的系统 |
| 需求分析 | 把模糊的目标变成清晰的Spec |
| 项目管理 | 把大目标拆成可执行的小任务 |
| 复杂性管理 | 把复杂系统拆成可理解的模块 |

### 12.1 核心观点回顾

| # | 观点 | 重要性 |

| 3 | 人类最应关注Goal，而非代码 | ⭐⭐⭐⭐⭐ |
| 4 | Spec比多Agent协作更重要 | ⭐⭐⭐⭐ |
| 5 | 大力出奇迹，省Token做不出好软件 | ⭐⭐⭐⭐ |
| 6 | 模型智商是决定性因素 | ⭐⭐⭐⭐ |
| 7 | 软件形态从静态UI走向意图驱动 | ⭐⭐⭐⭐ |

### 12.2 行动指引

**对于软件工程师**：
1. 从"写代码"转向"定义目标"——想清楚你要什么比怎么实现更重要
2. 学会写好Spec——这是人类在Agent时代最核心的技能之一
3. 不要省Token——想做好软件就要接受大力出奇迹

**对于技术决策者**：
1. 评估Agent能力时，用长程复杂任务测试，而不是简单任务
2. 投资方向：往"极度贴近人"或"极度贴近机器"两端走
3. 关注"Agent体工学"这个新赛道

**对于创业者**：
1. 机会在两端：更好的交互体验 / 更纯粹的基础设施
2. Skill是过渡方案，思考下一代抽象是什么
3. 你的领域Know-How可以蒸馏成"Stack"来变现
