### 1.2 项目定位

> **"Build better enterprise architecture through structured governance, vendor procurement, and design review workflows."**
> "通过结构化治理、供应商采购和设计评审工作流，构建更好的企业架构。"

ArcKit将企业架构治理从散落的文档转化为系统性、AI辅助的工作流，解决传统架构治理的三大核心问题：

| 问题 | 传统方案 | ArcKit方案 |
|------|----------|------------|
| **文档散落** | Word/Confluence/PPT各自为政 | Git版本控制+统一模板 |
| **治理执行不一致** | 依赖个人经验和意愿 | 标准化流程+自动检查 |
| **可追溯性丢失** | 需求与设计脱节 | 双向追踪矩阵 |

## §2 问题与解决方案

### 2.1 传统企业架构的困境

```
┌─────────────────────────────────────────────────────────────┐
│                    传统企业架构治理的问题                         │
│                                                              │
│  ❌ 文档散落                                                 │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐       │
│  │ Word    │  │Confluence│  │  PPT    │  │ Excel   │       │
│  │ 文档    │  │  文档   │  │  演示   │  │ 表格    │       │
│  └─────────┘  └─────────┘  └─────────┘  └─────────┘       │
│  → 版本混乱、找不到最新、冲突覆盖                              │
│                                                              │
│  ❌ 治理执行不一致                                           │
│  → 依赖架构师个人经验和意愿                                  │
│  → 不同项目差异巨大、无法标准化                                │
│                                                              │
│  ❌ 供应商选择偏见                                           │
│  → 商务关系影响大于技术评估                                  │
│  → 缺乏系统性评分框架                                        │
│                                                              │
│  ❌ 可追溯性丢失                                             │
│  ┌─────────┐      ┌─────────┐      ┌─────────┐            │
│  │ 需求    │ ──/── │ 设计    │ ──/── │ 测试    │            │
│  └─────────┘      └─────────┘      └─────────┘            │
│  → 需求变更不知道影响哪些设计                                 │
│                                                              │
│  ❌ 文档过时                                                 │
│  → 设计与实现不符、文档无人维护                               │
└─────────────────────────────────────────────────────────────┘
```

## §6 Wardley Mapping

### 6.1 什么是Wardley Mapping

Wardley Mapping是一种战略规划工具，通过在演化轴（Genesis → Custom → Product → Commodity）上定位组件来揭示竞争动态。

```
↑
y                      ■ 核心服务
轴                     
|                 ■     ■  通用能力
演                     ◆         ■ 组件
化                     ◆    ◆        
轴                     ◆               
|      ●              ◆              ■ 基础设施
|                    ◆                    
|─────────────────────────────────────────▶
  Genesis   Custom   Product   Commodity
```

### 6.2 ArcKit中的Wardley Mapping

**`/arckit-platform-design`** 自动生成Wardley图：

```mermaid
wardley
    title 数字平台Wardley图
    axis beta
    genesis[定制开发] visible 7
    custom[定制解决方案] custom 6
    product_a[商业SaaS A] product 5
    product_b[商业SaaS B] product 5
    commodity_iaas[IaaS] commodity 4
    commodity_network[网络/CDN] commodity 3
    commodity_power[电力] commodity 2
    genesis --> custom
    custom --> product_a
    custom --> product_b
    product_a --> commodity_iaas
    commodity_iaas --> commodity_network
    commodity_iaas --> commodity_power
```

## §8 最佳实践

- [ ] Phase 0: 创建项目结构和计划
- [ ] Phase 1: 建立架构原则（获批）
- [ ] Phase 2: 完成利益相关者分析

- [ ] Phase 4: 完成SOBC（获批后再继续）
- [ ] Phase 5: 定义完整需求

### 8.2 常见陷阱

| 陷阱 | 避免方法 |
|------|----------|
| 跳过Phase 2-4直接做需求 | 必须在SOBC获批前完成 |
| 需求过于详细 | 聚焦高优先级，迭代细化 |
| 忽视NFR | NFR决定架构约束 |
| 供应商锁定 | 保留退出策略 |
| 文档过时 | 建立文档更新机制 |
