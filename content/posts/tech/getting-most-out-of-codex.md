# Getting the Most Out of Codex：AI coding agent 的能力边界与工作范式

## 一句话核心判断

Codex 正在从一个「代码助手」演变成「计算机工作操作系统」——它真正解决的不只是帮程序员写代码，而是把一切需要操作计算机完成的工作（执行 shell、浏览网页、调用 API、响应事件、触发自动化）都纳入同一个持久化、可协作的上下文框架。

---

## 总览：Codex 的能力地图

```
┌─────────────────────────────────────────────────────────────┐
│                     Codex 工作层                            │
├─────────────────────────────────────────────────────────────┤
│  持久线程层     │  上下文跨 session 保持，Command+Pin 快速访问  │
│  语音 + 操控层  │  语音输入 + Steering(干预) + Queuing(排队)   │
│  工具扩展层     │  $browser · @chrome · @computer · MCP       │
│  自动化层       │  定时任务 + Thread Automations + Goals      │
│  产物层         │  代码 + 文档 + 演示文稿 + 任意格式工件         │
└─────────────────────────────────────────────────────────────┘
```

三层向外延伸的路径：
- `$browser`：侧边栏内嵌浏览器，适合页面审查和标注
- `@chrome`：使用用户已登录的 Chrome 状态，适合依赖用户身份的工作流
- `@computer`：通过桌面 GUI 操作，适合只能通过图形界面完成的任务

## 1. 持久线程：从「问完就跑」到「持续工作空间」

传统 AI 对话的问题是：每次对话都是全新的上下文，重复的背景信息需要反复提供。Codex 的「持久线程」彻底改变了这一点。

### 1.1 Pin Thread 的实际用法

Pin 起来的线程是一个长期工作空间，能保留：
- 过往的决策记录
- 用户偏好设置
- 工作进行中的上下文

典型适用场景：

| 线程类型 | 用途 | 价值 |
|---------|------|------|
| Chief of Staff | 综合助理，统筹信息流 | 减少每次重新解释背景 |
| Release Thread | 发布流程跟踪 | 每次回来继续上次进度 |
| Documentation Review | 文档审查 | 持续追踪修改意见 |
| External Monitoring | 外部监控 | 定期检查并报告变化 |

Command+1 到 Command+9 可以直接跳转到固定的 Pin Thread，这是让工作流真正落地的关键。

### 1.2 与传统聊天的本质区别

不是「更长的对话」，而是一个**可中断、可继续、有记忆的工作会话**。用户离开后回来，Codex 知道之前做到了哪里、什么决策已经做过。

## 2. 语音输入：捕获思维的原始版本

语音输入的价值在于捕获「思维压缩前」的状态。很多想法说出来自然但打出来费劲：

> "I think someone named Ben mentioned this in Slack. I do not remember the details. Please go look."

这种模糊的起点，对一个能搜索、收集信息、报告结果的 Agent 来说，往往已经足够。

另一个高效场景：**任务还没完全成型时的 2-3 分钟思维倾倒**。写下来的东西往往比口述更规整，但口述能保留思维的多样性和未完成的线索——这些在最终产出中反而可能是价值。

**适用判断**：如果你要说的内容自然流畅但打字麻烦 → 用语音。如果你需要精确的格式和措辞 → 直接打字。

## 3. Steering 与 Queuing：控制正在发生的工作

这是两个不同方向的控制动作：

### Steering（转向）
**在当前任务未完成时插入新方向**，相当于「停，现在先做这个」。

典型场景：Codex 在审查网站，方向走偏了，用户直接在侧边栏标注：
- "make this smaller"
- "the spacing between these two elements feels off"
- "this copy is wrong"

Steering 的关键在于**在错误完成前拦截**，避免浪费算力和时间。

### Queuing（排队）
**在当前任务结束后加入下一个任务**，不打断正在做的事。

典型场景：
> "Once the work is done, send the preview link to the reviewer in Slack."

**核心区别**：Steering 改变「现在做什么」，Queuing 改变「接下来做什么」。两者都让人保持对工作的近程参与。

## 4. 工具扩展：从代码到全计算机操作

### 4.1 三层工具的能力边界

```
$browser    → 侧边栏内嵌浏览器，Codex 可以检查和标注 Web 表面
@chrome    → 用户已登录的 Chrome，适合依赖用户身份的工作流
@computer  → 桌面 GUI 操作，适合那些「没有 API 只有图形界面」的任务
MCP Server  → 扩展到 Slack、Gmail、Calendar 等，让信息流和问题解决连在一起
```

MCP（Model Context Protocol）Servers 是这里的关键：它把 Codex 能操作的边界从「代码」扩展到「日常工作流出现的任何形式的任务」。

### 4.2 Skills：将工作流变成可复用工具

当一个工作流被验证有用，就把它打包成 Skill。这样 Codex 不需要每次从头学，Skill 可以直接执行。

这解决了一个常见问题：Agent 学到的流程如何固化下来、而不是每次都重新摸索。

## 5. 移动端：从「必须在桌前」到「随时接入」

Codex 移动端改变的不是功能，而是**任务的连续性**：

- 在 Mac 上开始一个任务（文件、权限、本地环境都在）
- 出门后继续：在手机上查看进度、批准下一步、调整方向
- 回到桌前，工作环境和上下文都在原地等你

这解决的是「长时间任务」的执行悖论：以前这类任务需要持续在场，现在可以分段执行而不丢失上下文。

## 6. Automations：让工作在人不在时继续

### 6.1 两种自动化的选择逻辑

| 自动化类型 | 触发条件 | 适用场景 |
|-----------|---------|---------|
| **定时自动化**（Scheduled Automation） | 按时间触发，任务从全新的 workspace 开始 | 每日报表、定期仓库检查 |
| **线程自动化**（Thread Automation） | 返回同一个活跃线程，保留运行上下文 | 需要承接上次的进度、需要持续追踪 |

### 6.2 Thread Automation 的实际价值

线程自动化是心跳风格的定时唤醒，但返回的是一个有上下文的工作会话。

**Chief of Staff 线程示例**：

```
每 30 分钟执行：
1. 检查 Slack 和 Gmail 未回复的消息（需要我关注的部分）
2. 帮我排列优先级
3. 如果有人问我问题，尽可能深入研究并起草回复草稿（不发送）
```

用户回来时，收集上下文的过程已经完成，人只需要决定「发送什么」。

### 6.3 反馈循环场景

一个动画工作流的例子：

1. 审查者在 Slack 分享了一个视频评论
2. 线程自动化定期检查线程
3. 有新评论时触发 Codex 渲染新版本
4. 在同一线程回复并 tag 审查者

如果一个集成不能完成最终上传，桌面自动化可以通过 GUI 完成最后一步——整个链路跨越 Slack（反馈）、codebase（渲染）、desktop（上传）。

## 7. Goals：给 Agent 一个可推进的终点线

**Goals 最有价值的场景**：任务有真实可衡量的完成线，Agent 可以持续向其推进。

**强目标 vs 弱目标**：

```
❌ 弱目标：实现这个 Markdown 文件里的计划

✅ 强目标（可验证）：
  把内部工具从 Python 迁移到 Rust
  → 设置新目录
  → 定义目标（unit tests pass 即为完成）
```

### Goals 的核心组合

```
Goal = 持续执行 + 验证器
用户定义：outcome（结果）+ stopping condition（停止条件）+ 
         signal（判断是否更接近目标）
```

有效验证器的例子：

| 验证器类型 | 适用场景 |
|-----------|---------|
| 测试套件 | 功能迁移、代码重构 |
| Benchmark | 性能优化任务 |
| Bug 复现 | 问题修复验证 |
| 端到端工作流 | 完整流程必须持续通过 |

**Ambition matters, but without verification it's just a wish.** —— 有愿景没有验证，只是愿望。

## 8. 侧边栏：让产物在对话中原地审查

侧边栏的核心价值是**把工件留在对话的上下文里**，不需要导出和切换窗口。

适合侧边栏的四种核心工作：

```
1. Inspect artifacts    → 直接查看输出产物
2. Annotate changes     → 标注需要修改的地方
3. Operate web surfaces → 在 Web 表面直接操作
4. Review changes       → 审核代码或文档修改
```

输出可能是代码，也可能是演示文稿、PDF、网页、表格——Codex 在工作过程中产生的任何格式的工件，都可以在侧边栏直接查看和操作。

## 9. 采用建议：谁先上、谁等等、从哪开始

### 适用顺序

**第一批（现在就用）**：
- 开发者：日常代码审查、Bug 修复、文档写作
- Technical Lead：代码审查、架构决策追踪
- 内容创作者：技术文档、演示文稿生成

**第二批（场景成熟后）**：
- PM/运营：使用 Chief of Staff 线程管理信息流
- 团队：建立 MCP 连接 Slack/Gmail/Calendar 的工作流
- 移动工作者：通过移动端接入长时间任务

### 从哪里开始

1. **先建立 Pin Thread**：选一个最常用的工作流（代码审查、发布跟踪），把持久上下文用起来
2. **语音输入替代打字**：从「模糊起点」场景开始，体验思维直接转物件的效率
3. **尝试 Steering**：在 Codex 走偏时拦截，而不是等它完成后再重来
4. **建立第一个自动化**：选一个周期性任务（每日报告、仓库检查），体验不在场时工作继续的感觉

### 不适合的场景

- 完全没有计算机操作知识基础的用户（Codex 是工具，不是替代思考）
- 需要精确控制、每步验证的审计类任务（当前 Agent 的不确定性仍存在）
- 实时性要求极高、延迟成本极大的交易类操作

## 总结：Codex 演进的三个阶段

| 阶段 | 核心能力 | 用户状态 |
|------|---------|---------|
| **第一阶段：代码助手** | 生成代码、修复 Bug、解释代码 | 人问，AI 答 |
| **第二阶段：上下文管理器** | 持久线程、跨 session 记忆、Pin Thread | AI 持续工作，人周期性介入 |
| **第三阶段：工作流操作系统** | MCP 扩展、Automations、Goals、跨工具协作 | 人定义目标和边界，AI 自主推进 |

Codex 的核心变化是：**从「工具」变成「操作系统」**——它不再只响应单个请求，而是管理持续的上下文、执行跨工具的工作流、在用户不在时继续推进目标。

理解这一点的关键在于：Codex 真正在替代的不是「编程」，而是「操作计算机完成工作」这件事本身。编程只是最先成熟的那个场景。
