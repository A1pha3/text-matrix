# YouTube 视频深度解读工作流

将 YouTube 视频转化为可发布的中文深度解读文章，核心原则是先取证，再写作。

## 0. 触发与路由

### 触发信号

用户消息包含以下任一形式即触发：
- `https://www.youtube.com/watch?v=...`
- `https://youtu.be/...`
- `https://www.youtube.com/embed/...` 或 `/shorts/...`
- B站链接 `https://www.bilibili.com/video/...`
- 纯视频ID（如 `dQw4w9WgXcQ`）
- 视频标题（如 "AI Agents, Clearly Explained"）

### 模式路由

| 模式 | 触发信号 | 输出 |
|------|----------|------|
| 分析 | "看看这个视频值不值得写""帮我分析这个视频" | 视频信息卡 + 选题建议 |
| **写作** | "根据这个视频写文章" / **只有链接（默认）** | 完整文章草稿 |
| 发布 | 明确要求"发布到 text-matrix / 飞书 / 更新索引" | 文章文件 + 发布结果 |

**默认行为**：只有链接，或链接+短指令（"写文章""深度解读""整理成笔记"）→ 直接进入写作模式，不追问。只有明确要求时才进入发布模式。只有当用户明确要求"只分析不写""只给摘要"时，才覆盖默认写作动作。

收到非标准链接（playlist、channel）时，先归一化为视频播放页面再继续。

## 1. 适用边界

### 适合

- YouTube/B站 视频链接、视频ID，或要求分析某个视频内容
- 目标产物是中文视频深度解读、视频笔记、学习指南
- 目标站点是 text-matrix 或兼容 Hugo Frontmatter 的内容库
- 视频内容为：技术教程、知识分享、课程讲座、产品评测、行业分析等

### 不适合

- 纯娱乐、vlog、新闻现场报道
- 没有视频证据、只有模糊想法的营销文案
- 要求伪造视频数据、观看量、评论内容或频道信息

## 2. 执行流程

### Step 1: 视频取证

先从 YouTube/B站 页面获取视频证据，再决定文章结构。

非标准链接（playlist、channel）先提取：原始链接 → 归一化后的播放链接 → 视频ID。

**取证工具（按优先级排序）**：

1. `mcp__web-reader__webReader` — 首选。两个平台均返回结构化 metadata（title / description / publishedTime / keywords 等），无需解析 HTML
2. `WebFetch` — 备选。当 webReader 超时或返回异常时使用
3. `mcp__web-search-prime__web_search_prime` — 补充搜索。用于查找视频评论摘要、相关讨论、频道背景等 webReader 无法获取的信息

**YouTube 取证字段**（从 metadata 提取）：

| 字段 | metadata 路径 | 示例 |
|------|---------------|------|
| 标题 | `title` | `"Rick Astley - Never Gonna Give You Up (Official Video) (4K Remaster)"` |
| 频道 | `name[1]` | `"Rick Astley"` |
| 发布时间 | `datePublished` | `"2009-10-24T23:57:33-07:00"` |
| 时长 | `duration` | `"PT3M34S"` → 3分34秒 |
| 观看/点赞 | `userInteractionCount` | `["19032350", "1767368233"]` |
| 标签 | `keywords` | `["rick astley", "Never Gonna Give You Up", ...]` |
| 描述 | `description` | 视频描述全文 |
| 分类 | `genre` | `"Music"` |

**B站取证字段**（从 description 文本解析）：

| 字段 | 提取方式 | 示例 |
|------|----------|------|
| 标题 | `title` | `"【官方 MV】Never Gonna Give You Up - Rick Astley"` |
| UP主 | `metadata.author` | `"索尼音乐中国"` |
| 发布时间 | `datePublished` | `"2020-01-01 07:43:23"` |
| 播放量 | description 中 `播放量` 后的数字 | `87522956` |
| 弹幕量 | description 中 `弹幕量` 后的数字 | `122616` |
| 点赞数 | description 中 `点赞数` 后的数字 | `2314953` |
| 投硬币 | description 中 `投硬币枚数` 后的数字 | `989011` |
| 收藏数 | description 中 `收藏人数` 后的数字 | `1211965` |
| 转发数 | description 中 `转发人数` 后的数字 | `365833` |
| 相关视频 | `content` 中推荐区 | 标题 + UP主 + 播放量 |

**B站独有注意事项**：
- B站无章节时间戳功能；如有分段需求，从描述或评论区提取手动标注的时间点
- B站无 `duration` 字段；时长需从描述文本或用户补充获取
- B站分区（如"科技→计算机"）不直接体现在 metadata 中；需要时从页面内容推断
- 投硬币、收藏、转发是 B站独有的互动指标，写入文章时保留这些数据以体现平台特色

**两个平台均无法自动获取的字段**：订阅数/粉丝数、评论内容、字幕/transcript、视频章节（YouTube）。这些需用户补充或标记为"未知"。

**按需补充**：

- 热门评论内容（用 `web_search_prime` 搜索视频讨论）
- 相关视频推荐（同系列或同话题）
- 字幕/transcript（描述不足时，用搜索查找第三方字幕源）
- 频道/UP主其他热门视频

取证原则：描述足够时不必深挖；描述不足时通过搜索补证；无法确认的信息标记为"未知"，禁止补写。

### Step 2: 选题定位

根据证据判断文章主轴，只保留一个主叙事。

| 优先级 | 视频特征 | 文章方向 |
|--------|----------|----------|
| 1 | 知识密度高、干货多 | 核心要点提炼 + 深度解读 |
| 2 | 实操性强、有步骤演示 | 教程型笔记 + 实战指南 |
| 3 | 观点新颖、有独特见解 | 观点解析 + 思考延伸 |
| 4 | 内容较新但信息有限 | 视频导读 + 观看建议 |

选题输出：目标读者 / 一句话标题方向 / 为什么值得写 / 本文不覆盖什么

### Step 3: 生成文章草稿

进入写作时，加载 [../cn-doc-writer/SKILL.md](../cn-doc-writer/SKILL.md)。

文章定位：分类=视频精读，文风=专业克制可验证，目标=帮助读者判断视频价值并提取关键信息。

推荐结构：

1. 视频概览（标题、频道、时长、观看量、发布时间）+ **原视频链接**（必须可点击跳转）
2. 学习目标或读者收益
3. 视频核心内容梳理（按时间线或主题）
4. 关键要点提炼（3-5个核心观点）
5. 重要时间戳/章节索引
6. 适用观众、优势与局限
7. 常见问题或观看建议
8. 总结与延伸阅读

结构不是硬性模板；信息不足时优先缩短而非补写空章节。

**正例**（开头 blockquote 格式）：

```markdown
> **目标读者**：想系统掌握 AI Agent 技术的开发者
> **核心问题**：从单体 Agent 到 Multi-Agent 的架构如何演进？
> **难度**：⭐⭐⭐⭐ | **来源**：YouTube @NickSaraev, 2026-04-28
```

**反例**（避免）：

```markdown
本文将为你详细解析这个视频的全部内容，让你从零开始掌握...
```
→ 禁止"本文将"开头，禁止空洞的"从零开始"承诺。直接进入核心信息。

### Step 4: 三轮事实校验

草稿完成后按顺序检查，任意一轮失败先修正再继续：

1. **事实一致性**：标题、频道名、数据、命令、时间戳与视频页面一致
2. **术语一致性**：同一概念的中英文叫法统一，首次出现保留原文+中文括注
3. **证据边界**：删除所有无来源的统计数据、观点引用、预测性结论

### Step 5: 生成 Hugo Frontmatter

加载：
- [../hugo-writer/SKILL.md](../hugo-writer/SKILL.md)
- [references/frontmatter-template.md](references/frontmatter-template.md)

约束：categories 必须是 `["视频精读"]`，tags 2-5 个精准名词，date 使用当前北京时间且格式完整，description 50-100 字纯文本。

写作模式默认产物：完整文章 + 合法 Frontmatter（除非用户明确要求只输出大纲）。

### Step 6: 发布与落库

仅在用户明确要求时执行。发布前确认：文件路径、工作区、是否同步飞书、是否更新索引。

如需更新索引，加载 [references/publish-index-template.md](references/publish-index-template.md)。

发布结果回报：文章路径 / git commit hash 或未提交 / 飞书链接或未同步 / 索引更新状态。

## 3. 输出格式

### 分析模式

最小产出清单：
1. 视频信息卡（标题、频道/UP主、时长、观看量、发布时间、原视频链接）
2. 值得写的角度（1-3 个方向）
3. 推荐标题与文章结构
4. 证据不足项（明确标注哪些字段缺失）

### 写作模式

最小产出清单：
1. 合法 Hugo Frontmatter（categories: ["视频精读"]）
2. 完整 Markdown 文章正文（含原视频可点击链接）
3. 不确定信息显式标注（如 `> 注：订阅数未能从页面获取，标记为未知`）

### 发布模式

最小产出清单：
1. 写作模式全部产物
2. 文章文件路径
3. git commit hash 或未提交状态
4. 飞书链接或未同步原因
5. publish-index 更新状态

## 4. 必须与禁止

### 必须

- 先取证再写作；禁止按视频名想象内容
- 只基于页面、描述、评论、字幕等可验证材料写作
- 信息不足时收缩范围，不扩写空洞章节
- 文章正文必须包含原视频可点击链接，方便读者跳转观看
- 默认中文输出，保留英文术语+首次中文括注
- 发布前完成三轮事实校验

### 禁止

- 伪造观看量、点赞数、订阅数、评论内容
- 把视频中没有的命令/步骤/结论写成"视频中提到"
- 未获指令时执行 git push、创建飞书文档、更新索引
- 依赖固定机器路径；始终以当前工作区或用户指定路径为准
- 凑结构补写"最佳实践""FAQ""延伸阅读"空章节

## 5. 异常处理

| 情况 | 处理 |
|------|------|
| 只有标题，没有链接 | 按标题搜索分析；无法确认时要求补充链接 |
| 描述很短 | 进入评论、字幕、章节补证 |
| 视频不可访问 | 告知证据不足，只基于用户提供材料写作 |
| 内容明显实验性/未完成 | 改为"视频导读/内容预览"，不写成完整教程 |
| 用户只要摘要 | 输出视频解读，不强制生成完整文章 |

## 6. 输出前自检

- 每个关键判断能否回溯到视频证据
- 标题、摘要、正文是否围绕同一主叙事
- 数据、观点、结论是否来自可验证材料
- 是否错误进入了发布模式
- Frontmatter 是否符合 Hugo 规范（categories: ["视频精读"]）
- 正文是否包含原视频可点击链接
