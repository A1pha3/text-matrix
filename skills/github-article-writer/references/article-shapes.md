# 仓库证据到文章形态的路由规则

> 适用于 github-article-writer 在选题定位阶段判断文章形态。先定形态，再定结构，再写正文。

---

## 先判断这篇文章是什么

不要把所有 GitHub 仓库都写成同一种“项目介绍 + 安装 + 总结”模板。更稳的做法是先判断文章形态。

默认从下面 5 类里选 1 类作为主形态：

1. **项目导读**：仓库有价值，但信息不足以支撑深度机制拆解。
2. **快速上手**：安装、示例、CLI、操作路径是读者最关心的部分。
3. **原理拆解 / 架构分析**：仓库真正的价值在系统设计、机制拆分或工程取舍。
4. **源码分析**：实现细节、模块协作和代码结构是最值得写的部分。
5. **实验性评估**：仓库较新、资料有限，但某个方向值得阶段性观察。

一篇文章只能有一个主形态。必要时可以借少量副形态补充，但不能把 5 类都塞进同一篇文章。

---

## 路由信号

| 看到什么证据 | 更可能的文章形态 |
| ------ | ------ |
| README 很完整，但主要是定位、能力、示例，源码细节不多 | 项目导读 |
| 安装命令、配置步骤、demo、CLI 用法占比高 | 快速上手 |
| 仓库里有多层子系统、并行机制、记忆层级、调度链路、benchmark 解释 | 原理拆解 / 架构分析 |
| 目录结构清晰，核心模块边界明确，代码实现本身有代表性 | 源码分析 |
| 仓库信息不多，但思路新，适合写“阶段性观察与边界” | 实验性评估 |

出现下面 3 类信号时，默认优先考虑“原理拆解 / 架构分析”，而不是普通教程：

- 项目价值主要来自一套新的系统分层或执行模型。
- README 有 benchmark、trade-off、memory、planner、runtime、compiler、agent loop 等结构性内容。
- 仓库里存在两套以上容易混淆的并行机制，需要先拆边界再展开。

---

## 输出契约

进入正文前，至少先明确下面 5 件事：

- 主形态：这篇文章到底是导读、教程、架构分析、源码分析还是实验性评估。
- 核心判断：这篇文章最想帮助读者建立什么判断。
- 证据集合：这个判断主要靠 README、源码、目录结构、benchmark、release 还是示例支撑。
- 目标读者：谁最适合读这篇文章。
- 不覆盖什么：这篇文章刻意不展开哪些部分。

如果这 5 件事说不清，正文大概率会重新滑回通用模板。

---

## 原理拆解 / 架构分析的额外动作

当主形态是“原理拆解 / 架构分析”，或文章高度依赖 benchmark 与系统分层时，必须额外加载：

- `../cn-doc-writer/references/blog-deep-dive.md`

并执行下面 5 条强制动作：

1. 开头先给判断，不先堆 Stars、Forks、版本号或 feature list。
2. 在前 20% 内给读者一张系统地图：图、表或总览段至少一种。
3. 至少补一个“请求 / 任务如何流过系统”的具体案例。
4. benchmark 段必须回答“测什么、反映哪部分系统、不能推出什么”。
5. 结尾给采用顺序、适用边界或决策建议。

---

## 常见反模式

- 把 README 的 feature bullets 直接扩写成文章主体。
- 同时想写教程、架构分析、源码分析，结果三者都不够深。
- 文章开头先报 Stars、License、安装命令，但迟迟不给判断。
- benchmark 只复述数字，不解释这些数字对应的能力边界。
- 本来是架构文章，却没有任何系统地图或任务流案例。

---

## 最小判断模板

写正文前，可以先在脑中形成这一组最小骨架：

```text
这是一篇：{文章形态}
本文核心判断：{一句话判断}
主要证据：{README / 源码 / benchmark / 目录结构 / 示例}
目标读者：{谁}
不覆盖：{哪些部分}
```

如果其中任一项含糊不清，先补证据，再动笔。
