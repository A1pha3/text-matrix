# 边界情况、错误恢复与反馈闭环

> 本文件包含 skill 适用边界、边界情况处理、错误恢复策略和反馈闭环机制。由 SKILL.md 按需加载。

---

## Skill 适用边界

| 适合 | 不适合 |
|------|--------|
| 中文技术文档 | 纯英文文档（无中文需求） |
| 教程、指南、API 文档 | 创意写作、营销文案 |
| 项目 README | 学术论文（有专门格式要求） |
| 技术博客 | 法律/医学等专业文档 |
| 英译中技术文档 | 中译英 |

---

## 边界情况处理

| 场景 | 处理策略 |
|------|----------|
| **超大文档（>5000 行）** | 按 H2 标题拆分为独立分片，逐片处理，最后合并验证 |
| **多语言混合（中日韩英）** | 日/韩文按中文空格规则处理，其他语言保持原文 |
| **HTML 内嵌标签** | `<details>` `<summary>` 等保留原样，仅翻译标签内文本 |
| **表格内长文本** | 表格单元格内中英文空格规则同样适用，但不强制段落长度 |
| **数学公式** | LaTeX 公式 `$...$` `$$...$$` 视为代码块，不检查不翻译 |
| **Mermaid/图表** | Mermaid 代码块内关键词不检查，但节点文本可翻译 |
| **波浪号围栏 `~~~`** | 与反引号围栏同等处理，闭合必须用同种字符且长度 >= 开启 |
| **嵌套代码块** | 变长围栏（如 `````）内部的短围栏（如 ` ``` `）不视为边界 |
| **链接锚点含中文** | URL 编码中文锚点，确保 `[链接](#中文标题)` 可跳转 |
| **无前置知识上下文** | AI 应主动询问技术栈和目标读者，而非凭空假设 |

## 错误恢复策略

文档编写/翻译过程中常见故障及恢复方式：

| 故障 | 症状 | 恢复策略 |
|------|------|----------|
| **术语表缺失** | `terminology.json` 不存在或损坏 | 回退到内置基础术语表（中英文空格规则仍适用） |
| **格式检查误报** | 代码块/URL 内的「中英无空格」被误报 | 确认 `should_skip()` 逻辑覆盖了该区域，若是新区域则标记为已知限制 |
| **翻译上下文不足** | 技术术语在原文中无解释 | 优先查术语表 → 搜索官方文档 → 保留英文并加译注 |
| **模板不匹配** | 文档类型不在四级模板中 | 使用最近似的模板，在输出中说明偏差 |
| **文档过长截断** | 单次处理超出上下文窗口 | 分片处理：按 H2 标题切分 → 逐片翻译/编写 → 合并 → 全局术语一致性检查 |
| **跨文档术语不一致** | 多文档体系中同一术语译名不同 | 以 terminology.json 为唯一数据源，跨文档翻译前先锁定术语表 |
| **跨文档链接失效** | 学习路径引用的文档不存在 | 完整模式下输出前检查所有内部链接指向的文件是否存在 |

## 反馈闭环机制

文档质量需要持续迭代。以下机制确保文档在实际使用中不断改进。

### 反馈收集触发点

| 触发点 | 收集方式 | 示例 |
|--------|----------|------|
| **文档交付时** | 末尾附反馈锚点 | `> 📝 本文档是否解决了你的问题？[是/否/部分]` |
| **用户提问时** | 记录文档未覆盖的问题 | 用户 FAQ 未命中 → 补充到文档 |
| **代码变更时** | 检查关联文档是否过时 | API 签名变更 → 触发文档同步 |
| **版本发布时** | 对照 Changelog 检查文档 | 新增功能 → 检查是否有对应文档 |

### 反馈分类与响应

```text
P0-紧急（24h 内修复）：
  - 技术错误（代码示例无法运行）
  - 误导性描述（可能导致数据丢失/安全问题）

P1-重要（1 周内修复）：
  - 缺失内容（常见操作无文档覆盖）
  - 过时信息（版本已更新但文档未同步）

P2-改进（下次迭代处理）：
  - 表述不清（能看懂但有歧义）
  - 缺少示例（有文字说明但无代码）

P3-建议（评估后决定）：
  - 新增内容请求
  - 格式改进建议
```

### 迭代工作流

```text
收集反馈 → 分类优先级 → 定位文档位置 → 修改文档
    ↓                                        ↓
记录到 Issue/看板                      重新运行评估评分
    ↓                                        ↓
跟踪修复状态  ←────── 验证通过 ←────── 评分 ≥ B 级？
```

> **规则**：每次 `optimize-cn-doc` 执行时，应检查是否有积累的 P0/P1 反馈未处理。
