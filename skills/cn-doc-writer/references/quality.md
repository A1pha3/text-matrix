# 质量评估与规则优先级

> 本文件包含量化评估标准、规则冲突优先级和质量检查清单。由 SKILL.md 按需加载。

---

## 规则冲突优先级

当多条规则同时适用于同一段文本时，按以下优先级处理（高优先级覆盖低优先级）：

| 优先级 | 规则 | 说明 |
|--------|------|------|
| **P0** | 代码块内不检查 | ` ``` ` 或 `~~~` 包裹的区域完全跳过，包括空格、标点、术语 |
| **P1** | 行内代码内不检查 | `` ` `` 包裹的内容不检查中英空格，如 `apiKey` |
| **P2** | LaTeX 公式内不检查 | `$...$` 和 `$$...$$` 视为代码，不检查不翻译 |
| **P3** | YAML frontmatter 不检查 | `---` 包裹的元数据区域跳过 |
| **P4** | URL/路径保持原样 | `https://api.example.com` 内不加空格 |
| **P5** | 术语表 > 通用规则 | 术语表标注 `keep_en: true` 的术语保留英文 |
| **P6** | 中英文空格规则 | 正文中中英文 / 中文数字之间必须加空格 |
| **P7** | 术语首次标注规则 | 每个术语首次出现时标注中文翻译 |
| **P8** | Markdown 换行保留 | 行尾恰好 2 个空格视为 `<br>`，不视为多余空格 |

### 实际冲突示例

| 场景 | 适用规则 | 结果 |
|------|----------|------|
| `const apiKey = "sk-..."` 在代码块内 | P0 | 不检查，保持原样 |
| 正文中写 `apiKey` | P1 | 行内代码内不加空格 |
| `使用 Docker 部署` | P6 | 中英之间有空格 ✓ |
| 首次出现 API | P7 | 写作 `API（应用程序接口）` |
| 术语表中 `Docker` 在 no_translate 列表 | P5 | 不翻译，不标注中文 |

## 量化评估标准

每篇文档按以下 5 个维度评分，总分 100 分：

| 维度 | 权重 | 评分标准 |
|------|------|----------|
| **结构性** | 20% | 标题层级正确、目录清晰、逻辑连贯、导航完整 |
| **准确性** | 25% | 技术内容正确、术语使用一致、代码可运行、链接有效 |
| **可读性** | 25% | 中英文混排规范、段落适中、排版舒适、自然表达、格式统一 |
| **教学性** | 20% | 有学习目标、解释"为什么"、学习元素自然融入、递进合理 |
| **实用性** | 10% | 示例贴近真实、常见问题覆盖、错误处理清晰 |

### 评分细则

```text
结构性（20分）：
  - 标题层级无跳跃（5分）
  - 有清晰的章节划分（5分）
  - 内容逻辑递进合理（5分）
  - 有目录/导航/前后文链接（5分）

准确性（25分）：
  - 技术描述无事实错误（10分）
  - 术语使用一致（5分）
  - 代码示例完整可运行（5分）
  - 链接有效、版本信息准确（5分）

可读性（25分）：
  - 中英文之间有空格（5分）
  - 正文用中文标点、代码保留英文标点（5分）
  - 段落与句长适中，不过度堆叠短句（5分）
  - 叙述自然，无明显模板腔、生成式转场、自我声明堆积（5分）
  - 列表/表格/代码块使用合理，不过度为凑结构而堆砌（5分）

教学性（20分）：
  - 有明确的学习目标（5分）
  - 核心概念解释了"为什么"（5分）
  - 包含练习/自测/路径等学习元素，且与正文衔接自然（5分）
  - 难度递进合理（5分）

实用性（10分）：
  - 示例来自真实场景（4分）
  - 包含常见问题解答（3分）
  - 有错误处理和排查指引（3分）
```

### 可读性量化指标（辅助参考）

- 平均句长：15-35 字/句为宜（>50 字应拆分）
- 段落密度：每段 3-8 行正文（不含代码块）
- 信息密度：每段引入不超过 2 个新概念
- 标题间距：两个同级标题间建议 50-300 字正文
- 代码文本比：教程类建议 30%-50%，参考类可达 60%

### 评分等级

| 等级 | 分数范围 | 含义 |
|------|----------|------|
| S | 90-100 | 专家级，可作为范例 |
| A | 80-89 | 优秀，满足专业标准 |
| B | 70-79 | 良好，有改进空间 |
| C | 60-69 | 及格，需要优化 |
| D | <60 | 不合格，需要重写 |

## 去 AI 味门槛（不新增维度）

自然表达不单独新增第六维，而是作为 **可读性 + 教学性** 的强制门槛处理。评分时同时看词面、句式和节奏，不只看有没有“本文将”。
完整信号、反例和替换方向见 `references/examples.md` 的「完整信号库」；本节只定义评分约束。

要拿到 A / S，尤其是满分 100，必须同时满足：

- 没有连续元话语、空泛抽象词、口号式判断或机械总结。
- 学习目标、阅读指引、自测问题等结构块确实服务当前内容。
- 列表、小标题和段落节奏不为整齐而牺牲信息密度。
- 去 AI 味后五维总分不下降，且没有删除事实边界、版本差异、错误条件或 caveat。

若上述门槛任一项未满足：

- `可读性` 上限封顶为 20/25。
- 总分不得超过 89 分。
- 不得评为 S 级。

辅助判断：删掉抽象词后，段落仍应留下明确对象、动作、限制或结果；否则回到 `references/examples.md` 按完整信号库重写。

## 分析型技术文章附加检查

适用于开源项目解读、架构分析、benchmark 解读、系统评测等文体；不适用于普通 API 文档或操作手册。

满足越多，越容易达到高完成度的专家级成稿：

- 开头先给判断，而不是先给背景材料、Stars、版本号或功能列表。
- 正文前 20% 给出总览地图：图、表或明确总览段至少一种。
- 当系统包含两套以上并行机制时，先拆边界，再展开机制细节。
- 至少有一个“任务如何流过系统”的具体案例，而不是全篇静态结构说明。
- benchmark 段解释“测的是什么、对应哪部分系统、不能推出什么”。
- 结尾给采用顺序、适用边界或决策建议。

若是分析型文章且以上 6 项中低于 3 项满足：

- `结构性` 上限封顶为 16/20。
- `实用性` 上限封顶为 7/10。
- 不建议评为 S 级。

### 发布门槛

| 等级 | 发布建议 |
|------|----------|
| S/A（≥80） | ✅ 可直接发布 |
| B（70-79） | ⚠️ 可发布，但建议先修复主要扣分项 |
| C（60-69） | ❌ 不建议发布，需执行 `optimize-cn-doc` 流程优化 |
| D（<60） | ❌ 禁止发布，需重写或使用 `write-cn-doc` 重新生成 |

> **强制规则**：当 AI 生成的文档评分低于 B 级（70 分）时，必须在输出中明确标注警告：
> `⚠️ 本文档评分 XX 分（C/D 级），建议优化后再发布。`

## 质量检查清单

```markdown
### 结构
- [ ] 标题层级正确（# > ## > ### > ####，无跳跃）
- [ ] 目录/导航完整
- [ ] 代码块语法正确且指定了语言
- [ ] 所有链接有效

### 内容
- [ ] 术语使用一致
- [ ] 每个核心概念都解释了"为什么"
- [ ] 代码示例完整可运行
- [ ] 包含常见问题解答

### 格式
- [ ] 中英文之间有空格
- [ ] 数字与中文之间有空格
- [ ] 中文语境用中文标点
- [ ] 列表格式统一（有序/无序不混用）

### 自然表达
- [ ] 无明显模板腔、生成式转场、过度自我声明
- [ ] 无抽象套话密集堆叠、二元口号句、高光总结句
- [ ] 标题与导语像正文自然长出来，而不是表单标签
- [ ] 列表没有为了凑结构而机械对称
- [ ] 去 AI 味后五维评分未下降，事实边界未被删除

### 教学性（标准/完整模式）
- [ ] 有明确的学习目标
- [ ] 包含练习 / 自测 / 路径等学习元素，且与正文衔接自然
- [ ] 难度递进合理
- [ ] 有进阶路径指引
```
