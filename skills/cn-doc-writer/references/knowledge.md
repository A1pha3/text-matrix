## 上下文管理策略（超长文档处理）

当文档超出单次上下文窗口（通常 >3000 行或 >80K 字符）时，按以下策略分片处理：

```text
分片处理流程：

1. 预扫描阶段（全局信息提取）：
   - 提取完整标题结构（H1-H4 大纲）
   - 提取所有术语首次出现位置
   - 统计代码块数量和语言分布
   - 生成「文档地图」供后续分片引用

2. 分片规则：
   - 按 H2 标题自然分片（每片 1 个 H2 章节）
   - 若单个 H2 章节仍超长 → 按 H3 二次分片
   - 每片必须携带：文档地图 + 术语表 + 当前片上下文摘要

3. 分片间上下文传递：
   - 前序摘要：上一片的最后 3 句话 + 关键结论
   - 术语状态：已标注的术语列表（后续片不重复标注）
   - 交叉引用：本片被其他片引用的锚点列表

4. 合并校验：
   - 术语全文一致性检查
   - 标题编号连续性检查
   - 交叉引用有效性验证
   - 段落衔接自然度审查（片接缝处）
```

**分片上下文模板**（每片开头必须包含）：

```markdown
<!-- 分片上下文 -->
<!-- 文档：{文档标题} -->
<!-- 当前分片：第 {N}/{总数} 片 —— {H2 标题} -->
<!-- 前序摘要：{上一片最后 3 句} -->
<!-- 已标注术语：{术语1, 术语2, ...} -->
<!-- 待处理交叉引用：{#锚点1, #锚点2} -->
```

## 增量更新模式

当用户要求更新已有文档（而非全文重写）时，使用增量模式。

**触发条件**：用户使用 "更新 XX 章节" / "修改 XX 部分" / "把 A 改成 B" / "修复 XX 问题" 等表述时自动启用。

````text
增量更新流程：

1. 变更分析：
   - 识别用户指定的变更范围（章节/段落/全局）
   - 判断变更类型：内容更新 / 结构调整 / 格式修复 / 术语统一

2. 影响评估：
   - 直接影响区：需要修改的段落
   - 间接影响区：引用了变更内容的其他段落
   - 术语波及：如果术语名变更，需全文替换

3. 最小化修改：
   - 只输出变更部分 + 上下文（变更前后各 3-5 行）
   - 使用 diff 格式标注变更：
     ```diff
     - 旧内容
     + 新内容
     ```
   - 保持未变更章节的原始格式不动

4. 变更报告：
   - 列出所有修改位置（行号/章节）
   - 说明每处修改的原因
   - 标注间接影响区是否需要人工确认
````

### 增量模式核心约束

```text
1. 禁止全文输出：
   - 绝对不要输出未变更的章节——即使"为了完整性"
   - 未变更章节用占位符标注：「（第 X 章 无变更，保持原样）」

2. 变更定位格式：
   - 每处变更必须标注精确位置：章节标题 + 行号范围
   - 提供足够上下文（变更前后各 3-5 行），但不输出整章

3. 变更影响边界：
   - 仅当变更导致术语表、交叉引用、标题编号变化时，才列出间接影响
   - 间接影响仅列出位置和原因，不输出修改后的全文

4. 合并指引：
   - 在变更报告末尾，给出合并指令
   - 如果用户使用代码工具，输出可直接 apply 的 diff/patch 格式
```

## 多模型适配提示

| 模型特征 | 适配策略 | 适用模型 |
|----------|----------|----------|
| **强指令遵循** | 直接使用完整指令 | Claude、GPT-4、Gemini Pro |
| **上下文窗口较小**（<32K） | 优先加载核心指令，按需懒加载 references | 部分开源模型 |
| **中文能力较弱** | 增加中文输出示例（Few-shot） | 英文为主的模型 |
| **易忽略格式约束** | 将格式约束移到 prompt 末尾（近因效应） | 较弱的指令遵循模型 |
| **术语翻译不稳定** | 每次请求显式附带相关术语表子集 | 所有模型（推荐） |

**按模型调整加载策略**：

```text
大窗口模型（>128K）：加载 SKILL.md 全文 → 按需加载 references
中窗口模型（32K-128K）：加载核心指令 → 按需加载模板/路径
小窗口模型（<32K）：仅加载核心指令 + 术语表子集，每次只处理一个章节
```

## 学习目标设计要点

根据 Bloom 认知层次理论，学习目标分为六层：**记忆 → 理解 → 应用 → 分析 → 评价 → 创造**。

**书写规范**：
- 用动词开头：「能够 XXX」而非「了解 XXX」
- 可验证：「写出符合 REST 规范的 API 端点」而非「熟悉 REST」
- 分级标注：基础目标（必掌握）/ 进阶目标（建议掌握）/ 专家目标（挑战）

## 认知负荷管理要点

| 策略 | 做法 |
|------|------|
| 分块呈现 | 一次只讲一个概念，每块 3-8 行 |
| 先概念后细节 | 先给一句话定义，再展开 |
| 类比优先 | 复杂概念先用生活类比，再给技术定义 |
| 渐进复杂度 | 从最简示例开始，逐步增加复杂度 |

## 专家思维传授要点

**四种核心思维模型**：

| 模型 | 适用场景 | 文档中体现方式 |
|------|----------|----------------|
| 分解-抽象-模式识别 | 系统设计 | 架构图 + 模块拆解 |
| 假设-验证-迭代 | 调试排查 | 故障排查流程图 |
| 边界条件思考 | 代码审查 | 边界案例 + 错误处理 |
| 代价意识 | 技术选型 | 权衡表（获得 vs 失去） |

**解释"为什么"的框架**：问题背景 → 可选方案 → 选择理由 → 代价说明

## 体系框架

### 三条学习路径 × 四级进阶

| 路径 | 目标人群 | 核心问题 |
|------|----------|----------|
| **用户路径** | 使用工具/库的人 | "这个怎么用？" |
| **开发路径** | 参与开发的人 | "这个怎么实现的？" |
| **进阶路径** | 提升能力的人 | "为什么这样设计？" |

每条路径四个级别：Level 1⭐ 入门（模仿→执行）→ Level 2⭐⭐ 核心（理解→应用）→ Level 3⭐⭐⭐ 进阶（分析→优化）→ Level 4⭐⭐⭐⭐ 专家（创造→设计）

> 完整路径指南：[learning-paths.md](learning-paths.md)

### 术语与格式规范

**中英文混排规则**：

| 规则 | 正确 | 错误 |
|------|------|------|
| 中英文之间加空格 | `使用 Docker 部署` | `使用Docker部署` |
| 数字与中文之间加空格 | `共 3 个步骤` | `共3个步骤` |
| 行内代码前后加空格 | `运行 \`npm install\` 命令` | `运行\`npm install\`命令` |
| 括号内无需额外空格 | `功能（Feature）` | `功能（ Feature ）` |
| 中文标点用全角 | `这是一个示例。` | `这是一个示例.` |
| 英文/代码中标点用半角 | `config.json` | `config。json` |

**术语处理**：
- 首次出现：中文术语 + 英文原文 → _容器（Container）_
- 后续使用：统一使用中文或保持英文（如 Docker、Git 等专有名词不译）
- 术语表：[terminology.json](terminology.json)（单一数据源）

### 翻译规范

**核心原则**：准确 > 流畅 > 教学性增强

**关键规则**：
1. 不逐句直译，按段落重组为中文表达习惯
2. 代码块、命令、URL、文件路径保持原文
3. 同一术语全文保持一致翻译
4. 可适当补充背景解释和中文示例

### 文档体系治理指南

> 当项目文档超过 10 篇时，需要从「写文档」升级到「治理文档体系」。

#### 文档生命周期

```text
草稿(Draft) → 审核(Review) → 发布(Published) → 维护(Maintained) → 归档(Archived)

每篇文档应在 frontmatter 中标注：status / last_reviewed / owner
```

#### 跨文档一致性

| 检查项 | 工具/方法 | 频率 |
|--------|----------|------|
| 术语一致性 | `check_format.py` 全目录扫描 | 每次 PR |
| 内部链接有效性 | 检查所有 `[text](link)` 指向的文件是否存在 | 每次 PR |
| 版本号同步 | 文档中引用的版本号与代码一致 | 每次发版 |
| 过时内容检测 | `last_reviewed` 超过 90 天标记为待审 | 每月 |
| 孤立文档检测 | 没有被任何其他文档链接到的文档 | 每季度 |

#### 文档废弃流程

```text
1. 标记：顶部添加废弃告示 + 替代文档链接 + 计划移除日期
2. 重定向：更新所有引用链接
3. 归档：status → archived，移入 docs/_archived/
4. 移除：计划日期后删除，确认无残留引用
```

#### 文档与代码同步策略

| 场景 | 策略 |
|------|------|
| API 变更 | API 文档必须在同一 PR 中更新（CI 检查） |
| 配置项变更 | 更新对应的配置说明文档 |
| 新功能上线 | 至少更新 README 和对应的用户指南 |
| Breaking Change | 同时更新迁移指南和 Changelog |
| 依赖升级 | 更新安装和环境要求文档 |
